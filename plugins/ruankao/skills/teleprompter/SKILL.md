# 帮记词生成工作流

## 概述

帮记词不是摘要，也不是改写正文，而是背诵时的"逻辑路线图"。你将按标准格式为论文段落生成和维护帮记词。

## 目标

每条帮记词应帮助用户快速回忆：

1. 这一段先讲什么背景或问题。
2. 中间出现什么矛盾、原因、方法或调整。
3. 最后落到什么成果、反思或结论。

## 标准格式

统一使用 blockquote：

```markdown
> 记忆线：起点 -> 矛盾/原因 -> 方法/动作 -> 结果/结论
```

示例：

```markdown
> 记忆线：需求迫切 -> 旧架构失控 -> 录像事故暴露问题 -> 必须先建架构蓝图
```

## 生成原则

1. **一段一线**：一条帮记词只服务紧随其后的一个自然段或一个小论证单元。
2. **按逻辑走，不按词堆砌**：优先使用"背景 -> 问题 -> 原因 -> 方法 -> 结果"这类链路。
3. **短句优先**：每个节点控制在 4 到 12 个字左右，避免写成长摘要。
4. **抓关键词**：保留正文中的核心名词，如 ABSD、构件、复审、会话亲和、信媒拆分、性能取舍。
5. **便于口述**：读起来要像背诵提纲，避免"抛出、引出、承接"等写作过程词。
6. **不新增事实**：帮记词只能提炼正文已有信息，不能引入正文没有的数据、技术或结论。
7. **不改正文**：除非用户明确要求，正文论述、题干引用和参考材料不随帮记词一起改。

## 替换规则

### 已有 `> ...` 提示

如果正文里已有类似：

```markdown
> 挑战，挑战是什么，是耦合。
```

改为：

```markdown
> 记忆线：需求多而乱 -> 按技术层切错 -> 构件耦合 -> 接口联调受阻
```

### 已有括号提示

如果段落前是：

```markdown
(抛出问题，为什么引入ABSD，再点题和阐述论点)
```

也改为 blockquote 记忆线：

```markdown
> 记忆线：复杂挑战出现 -> 传统开发不足 -> 引入ABSD -> 交代论述范围
```

### 题干引用块

论文开头如果有题目要求、考试材料或大段引用，不要当作帮记词修改。

判断标准：
- 内容是论文题目、写作要求、评分要求、参考材料原文。
- 连续多行 blockquote，且不是正文段落前的提示。

这类内容保持原样。

## 推荐记忆线模板

### 项目背景段

```markdown
> 记忆线：业务需求增长 -> 旧系统暴露问题 -> 关键事故放大风险 -> 引出架构重构
```

### 方法引入段

```markdown
> 记忆线：决定重构 -> 引入方法 -> 对准非功能需求 -> 解决核心矛盾
```

### 问题分析段

```markdown
> 记忆线：现象复杂 -> 初始做法失误 -> 根因暴露 -> 影响扩大
```

### 解决方案段

```markdown
> 记忆线：找到根因 -> 调整划分/架构 -> 形成关键机制 -> 得到量化成果
```

### 结尾反思段

```markdown
> 记忆线：先报成果 -> 再讲局限 -> 总结取舍 -> 提出后续方向
```

## 工作流

### 第一步：定位候选提示

优先搜索：

```bash
rg -n "^>|^\\(|帮记|助记|记忆线|抛出|引出|挑战|成果|总结" "<论文路径>"
```

### 第二步：区分题干和正文提示

保留题干引用块，只处理正文段落前的提示。

正文提示通常满足：
- 位于 `## 摘要`、`## 项目背景`、`### ...` 等正文标题之后。
- 紧跟一个正文段落。
- 内容是写作提醒或背诵提醒，而不是正式论述。

### 第三步：读取上下文

每条帮记词至少阅读其后的一个自然段。

如果段落涉及承上启下，也阅读前一个标题或前一个段落，确保记忆线能接上上下文。

### 第四步：改写帮记词

按标准格式替换为：

```markdown
> 记忆线：A -> B -> C -> D
```

节点数量通常为 3 到 5 个。

### 第五步：检查一致性

修改后运行：

```bash
rg -n "^>|抛出|引出|挑战|成果|总结|帮记|助记" "<论文路径>"
```

确认：
- 正文提示都已统一为 `> 记忆线：...`
- 题干引用块没有被误改。
- 没有残留明显的过程词提示。

## 输出说明

完成后向用户说明：
- 修改了哪个文件。
- 帮记词已统一成什么格式。
- 是否保留了题干引用块。
- 是否发现残留未处理提示。

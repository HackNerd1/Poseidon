# 论文写作工作流

## 概述

你将从零开始，根据用户提供的软考论文题目（题干），按标准结构完成一篇约 2500 字的论文。流程覆盖：解析题干 → 选择/确认方法论 → 逐段生成 → 帮记词 → 字数校验。

## 工作流

### 第一步：解析题干

从用户提供的题目中提取三个核心要素：

1. **问题一（项目背景）**：通常要求"概要叙述你参与管理和开发的软件项目以及你在其中所承担的主要工作"——准备项目背景材料
2. **问题二（技术方法说明）**：通常要求"简要概述/分析 [某方法论/技术] 的 [核心概念/特性]"——确定需要用到的理论知识
3. **问题三（论点展开）**：通常要求"具体阐述你参与的项目如何基于 [某方法论/技术] 进行 [具体实践]"——确定论点展开的两到三个切入角度

### 第二步：加载参考材料

读取以下参考文档：

1. **references/写作技巧.md** — 各段字数规范、内容要求、句式模板、禁忌项
2. **references/范文.md** — 参考论文的结构、段落比例、用词风格、细节密度
3. **references/庭审系统PRD.md** — 项目事实数据（模块、性能指标、历史事故、用户角色等）

如果题目涉及以下方法论，额外加载对应 reference：
- 题干含 "ABSD" / "体系结构" / "软件架构设计方法" → 读取 `references/ABSD.md`
- 题干含 "ServiceMesh" / "服务网格" → 读取 `references/ServiceMesh.md`

### 第三步：确认项目信息

检查 PRD 和题干是否覆盖论文所需的全部项目信息：

- 项目名称、服务范围（如法院级别）、公司性质
- 核心技术栈及选型原因
- 具体业务模块（庭审管理、语音识别...而非"用户管理"）
- 量化数据（并发量、存储量、响应时间、成本等）
- 历史事故或痛点（增强说服力）

如有缺失 → 向用户提问补充。信息充足 → 进入第四步。

### 第四步：按结构逐段生成

严格按以下结构和字数要求生成，先写内容再加记忆线：

| 顺序 | 段落 | 目标字数 | 要求 |
|------|------|---------|------|
| 1 | 摘要 | 300 | 行业特色 + 技术视角 + 核心创新点 |
| 2 | 项目背景 | 500 | 业务意义（非技术栈罗列） + 核心模块 + 痛点 |
| 3 | 技术方法说明 | 400~500 | 理论概念系统性阐述，最低 400 字 |
| 4 | 论点展开一 | 500 | "错误→正确"句式 + 技术对比 + 具体数据 |
| 5 | 论点展开二 | 500 | 与论点一互补角度 + 场景细节 |
| 6 | 结尾 | 400 | 量化成效 + 批判性反思（非套路展望） |

**每段生成原则：**
- 用第一人称"我"/"笔者"写作
- 写业务独有的模块（庭审管理、会控信令…），禁止写权限管理、用户管理等通用模块
- 禁止罗列技术栈，要写为什么选它们
- 数据必须具体（"从 500ms 降至 80ms"，不是"显著提升"）

### 第五步：添加帮记词

为每个正文段落前添加标准格式记忆线：

```markdown
> 记忆线：起点 -> 矛盾/原因 -> 方法/动作 -> 结果/结论
```

记忆线节点 3~5 个，不新增论文中没有的事实。详细规则见 `teleprompter` 技能。

### 第六步：字数校验

运行字数统计脚本：

```bash
python scripts/count_chars.py <论文路径> "摘要=300" "项目背景=500" "技术方法说明=450" "论点展开标题1=500" "论点展开标题2=500" "结尾=400"
```

与规范比对，超出容差（±30）的段落需修剪直到达标。

### 第七步：总结

向用户汇报：
- 各段字数与目标对比
- 使用的 PRD 数据点清单
- 记忆线覆盖情况
- 建议后续用 `enhance-paper` 精细打磨的部分

## 关键规则

1. **先读后写**：必须先加载所有 reference 再动笔
2. **逐段生成**：按顺序一段一段写，每段写完后自检是否符合写作技巧要求
3. **数据驱动**：所有业务数据必须来自 PRD，禁止编造
4. **差异化优先**：写业务独有模块，拒绝通用模块；写批判反思，拒绝套路展望
5. **句子要具体**：用 "从 500ms 降至 80ms" 代替 "显著提升"
6. **人称统一**：全文用 "我"，一致到底

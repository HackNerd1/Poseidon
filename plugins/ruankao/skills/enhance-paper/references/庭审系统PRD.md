## 五、业务场景与负载特征

### 4.1 潮汐负载（论文核心数据点）

| 指标 | 平时 | 高峰期 |
|------|------|--------|
| 日均庭审量 | ~200 场 | ~2000 场（10x） |
| 高峰时段 | — | 工作日 9:00-11:00、14:00-16:00 |
| 计算需求 | 语音识别、笔录生成、录像存储 | 瞬时飙升至 10 倍以上 |

### 4.2 典型流程

```
书记员创建庭审 → 各方鉴权接入 → Janus 建立 WebRTC 媒体通道
→ 法官/书记员触发录制 → 语音识别实时生成笔录
→ FFmpeg 录制转码 → 录像归档 → 写入 MinIO → 更新 MySQL 索引 → 存储检索支持回放
```

## 七、技术架构速查

### 6.1 网络分区

```
外网区（终端接入） → 光闸区（双向数据光闸/安全代理） → 内网区（核心服务）
```

### 6.2 通信协议

| 类型 | 协议 | 说明 |
|------|------|------|
| 信令控制 | WebSocket + TLS | 低延迟双向 |
| 业务接口 | RESTful / HTTPS | 标准 API |
| 媒体传输 | WebRTC (DTLS-SRTP) | Janus SFU 转发 |

### 6.3 核心组件

| 组件 | 用途 |
|------|------|
| Spring Boot | 业务微服务框架 |
| Dubbo | 内网 RPC 调用 |
| Nacos | 服务注册与配置 |
| Spring Cloud Gateway | 统一 API 网关 |
| Redis | 缓存 + 分布式锁 |
| MySQL | 业务数据 + 录制索引 |
| MinIO | 对象存储（录像文件，EC 8+4 纠删码） |
| Janus | SFU 媒体服务器 |
| FFmpeg | 录制转码 |
| Kubernetes | 容器编排与弹性伸缩 |
| Docker | 容器化部署 |
| Prometheus | 自定义指标采集（HPA 预扩容） |

### 6.4 安全设计（按需引用）

| 层级 | 措施 |
|------|------|
| 通信层 | HTTPS / DTLS-SRTP、双向光闸隔离 |
| 认证层 | RBAC + JWT + MFA、Token 时效控制 |
| 数据层 | AES-256 加密、签名 URL（15min 有效）、SHA256 校验 + 水印 |
| 合规层 | 等保三级、ELK 审计、日志留存 ≥6 个月 |

## 八、历史事故（旧系统痛点）

- **2020 年录像丢失事故**：某次大规模庭审期间，存储节点 IO 过载导致录像写入链路崩溃，一批庭审录像丢失，直接危及司法证据链完整性。
- **扩容瓶颈**：旧耦合架构下扩容需同步采购计算+存储，交付周期 3~5 天。
- **性能不足**：SATA 盘随机读写延迟 >500ms，庭审页面卡顿。

## 九、关键技术选型依据（对比维度）

| 方案 | 适用场景 | 被否决原因 |
|------|----------|------------|
| SSD 缓存层（旧架构打补丁） | 数据有热点聚集 | 庭审材料访问分散，缓存命中率 <40%，效果有限 |
| 全闪存储 | 性能优先 | 年成本 >150 万，远超预算 |
| 三级分层存储 | **最终选择** | 兼顾 85% 热数据 <80ms + 70% 冷数据低成本 |
| 单 CPU 指标 HPA | 计算密集均匀负载 | IO 密集型服务漏扩，冷启动窗口致雪崩 |
| 双指标+预扩容 HPA | **最终选择** | CPU+QPS 双指标，排队超 5 场预扩容 |

# 论文优化工作流

## 概述

你将按照标准化工序对软考论文执行诊断和优化：加载参考材料 → 诊断原文 → 执行修改 → 字数校验 → 回写 PRD。

## 工作流

### 第一步：加载参考材料

同时读取以下参考文档，建立评判标准：

1. **references/写作技巧.md** — 各段字数规范、内容要求、句式模板、禁忌项
2. **references/范文.md** — 参考论文的结构、段落比例、用词风格、细节密度
3. **references/庭审系统PRD.md** — 项目事实数据（模块、性能指标、历史事故、用户角色等），论文中所有业务描述和技术数据必须与 PRD 一致
4. **目标论文** — 用户指定的待优化论文

### 第二步：检查上下文是否充足

对照写作技巧和 PRD，判断论文中涉及的项目信息是否足够支撑写作：

- 如果论文引用的业务场景、数据指标、历史事故等细节缺失或模糊 → **先向用户提问**补充项目信息
- 如果 PRD 中已有充足信息，论文可直接据实写作 → 进入第三步

常见需要确认的信息：
- 公司性质、项目服务范围（如法院级别）
- 历史事故的严重程度和具体时间
- 业务数据（并发量、存储量、响应时间等）
- 冷数据访问模式等场景细节

### 第三步：诊断原文是否符合要求

对照写作技巧逐项检查原文：

1. **结构完整性**：是否包含 `## 摘要` / `## 项目背景` / `## 正文`（含技术方法说明与 `###` 论点展开） / `## 结尾`
2. **字数合规**：用 `scripts/count_chars.py` 统计各段字数，与规范比对
3. **内容质量**：
   - 摘要是否突出行业特色 + 技术视角
   - 项目背景是否聚焦业务意义（而非罗列技术栈）
   - 技术方法说明是否水到 400 字以上
   - 论点展开是否使用"错误做法 → 正确做法"句式，是否有技术对比和具体数据
   - 结尾是否包含批判性反思（而非套路化展望）
4. **PRD 一致性**：论文中的项目描述是否与 PRD 事实相符
5. **笔误检查**：是否有明显拼写错误（如 "Kurbnets"）

### 第四步：执行修改

根据诊断结果决定操作类型：

| 诊断结果 | 操作 |
|---------|------|
| 缺少段落（如无摘要、无背景、无结尾） | **补充**符合规范的段落 |
| 内容不符合写作技巧要求 | **重写**该段落 |
| 字数超标 | **压缩**冗余内容，保留核心信息 |
| 字数不足 | **扩展**，增加数据/场景/对比维度 |
| 笔误 | **修正** |
| 完全符合要求 | 跳过，进入第五步 |

补充内容的优先级：
1. 先补充缺失的结构性段落（摘要、背景、结尾）
2. 再修复内容质量问题（泛泛而谈 → 具体数据）
3. 最后处理字数问题

### 第五步：字数校验

1. 运行字数统计脚本，将各段目标传入命令行参数，key 需与论文中 `##`/`###` 标题完全一致：
   ```bash
   python scripts/count_chars.py <论文路径> "摘要=300" "项目背景=500" "正文=450" "论点展开1标题=500" "论点展开2标题=500" "结尾=400"
   ```
2. 与下表比对：

| 段落 | 目标 | 容差 |
|------|------|------|
| 摘要 | 300 | ±30 |
| 项目背景 | 500 | ±30 |
| 正文（技术方法说明部分） | 400~500 | — |
| 论点展开（每段） | 500 | ±50 |
| 结尾 | 400 | ±30 |
| 正文合计 | ~2500 | — |

3. 超出容差的段落 → 反复修剪直到达标
4. 修剪时优先删除重复表述、过度修饰词，保留核心论据和数据

### 第六步：回写 PRD

检查优化后的论文是否产生了新的事实数据，如果有则更新到 `references/庭审系统PRD.md`：

- 新增的技术决策细节
- 新增的问题场景描述
- 新增的数据对比
- PRD 已有则不动，没有则补充

### 第七步：总结

向用户汇报：
- 各段最终字数与目标对比
- 核心改动列表（新增了什么、改了什么）
- 可以参考的 PRD 数据点

## 关键规则

1. **禁止写通用模块**：权限管理、用户管理、日志管理 → 减分
2. **必须写业务独有模块**：庭审管理、语音识别、笔录生成、录像归档 → 加分
3. **禁止罗列技术栈**：不要写 "用了 Spring Boot、Redis、Dubbo..."，要写为什么选它们
4. **必须有错误→正确的思维过程**：每个论点展开段落至少包含一次"我们错把...导致...后来发现...改为..."
5. **结尾必须批判**：不说"未来引入 AI/Service Mesh"的套路话，要说该方法引入的新问题
6. **数据必须具体**：不说"显著提升"，要说"从 500ms 降至 80ms"
7. **人称统一**：用"我"，全文一致

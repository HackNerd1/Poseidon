# 子 Agent 派发策略

> 此文档为 plan skill 的 L3 参考文档，在 SKILL.md Step 5 中按需加载。
> 定义了将详细计划中的任务派发给子 agent 实施的策略和模式。

---

## 一、派发模式

### 模式 1：串行派发

按依赖顺序逐个执行任务，每个任务一个子 agent。上一个完成后验证结果再启动下一个。

**适用场景：**
- 任务之间有强依赖链（T01 → T02 → T03）
- 共享状态/数据库 schema 变更（并发可能冲突）
- 下游任务依赖上游任务的输出作为输入

**流程：**
```
派发 T01 → 等待完成 → 验证 T01 结果 →
派发 T02 → 等待完成 → 验证 T02 结果 →
派发 T03 → ...
```

**优点：** 安全、可控、易于调试
**缺点：** 总耗时长（等于各任务之和）

### 模式 2：并行派发

无依赖的任务同时派发多个子 agent，全部完成后汇总结果。

**适用场景：**
- 任务之间无依赖关系（独立的功能切片、独立的文件）
- 各任务操作的文件/模块不重叠
- 需要加速完成多个独立任务

**流程：**
```
同时派发 T02, T03, T04（均只依赖已完成的 T01）
       ↓
等待全部完成
       ↓
汇总结果，检查冲突
```

**注意：**
- 写入同一文件的并行任务会冲突 → 应串行化或合并为一个任务
- 并行数建议 ≤ 5，过多会导致结果汇总成本高

**优点：** 速度快（总耗时 ≈ 最慢任务的时间）
**缺点：** 可能产生合并冲突，调试复杂

### 模式 3：仅当前任务

只派发下一个就绪的单个任务（依赖已满足的第一个任务）。

**适用场景：**
- 用户想逐步验证每个任务的产出
- 剩余任务之间有不确定性（下一步可能依赖当前任务的结果调整）
- 学习和理解型实施（边做边看）

**流程：**
```
派发下一个就绪任务 → 等待完成 → 验证 →
询问用户是否继续 → 派发下一个 → ...
```

**优点：** 最大控制力，可随时调整方向
**缺点：** 需要用户多次交互确认

### 模式 4：暂不实施

保留计划但不派发。

**适用场景：**
- 计划已生成，但用户需要手动实施部分或全部
- 等待外部条件（如 API key、数据准备）
- 计划作为参考文档，用户自行执行

## 二、派发方式选择决策树

```
任务是否都在等待外部条件？
├── 是 → 模式 4：暂不实施
└── 否 → 任务之间有依赖链吗？
    ├── 强依赖链（所有任务串行）→ 模式 1：串行派发
    ├── 无依赖（全部独立）→ 模式 2：并行派发（≤5 个）
    └── 部分依赖 → 分批：
        ├── 第一批：无依赖的就绪任务 → 并行派发（≤5 个）
        ├── 第二批：第一批完成后就绪的任务 → 并行派发
        └── ...
```

**推荐规则：**
- 就绪任务数 = 1 → 推荐模式 3（仅当前任务）
- 就绪任务数 = 2-5 且无依赖 → 推荐模式 2（并行派发）
- 就绪任务数 > 5 → 推荐分批并行，每批 ≤ 5
- 用户表示"快速迭代" → 推荐模式 1（串行，减少交互）
- 用户表示"谨慎验证" → 推荐模式 3（逐个确认）

## 三、子 Agent 派发规范

### 3.1 Agent 描述

简明标题，包含任务 ID：
```
Implement LOGIN-P1-01: Create user model
```

### 3.2 Agent Prompt 结构

```
根据详细计划文档 <plan-path> 中的 <TASK-ID> 任务：

**任务描述**：<从计划中复制>
**验收标准**：
- <从计划中复制>
**涉及文件**：<从计划中复制>
**依赖**：<已完成的前置任务及产出>
<如果项目有架构约束文档，附加：>
**架构约束**：请遵循 <constraint-doc-path> 中的规范

## 开发规范（必须遵守）

1. **关键代码必须添加注释**：
   - 公共函数/方法需说明用途、参数、返回值
   - 复杂逻辑（>10 行或含多个分支）需有行内注释解释思路
   - 非显而易见的决策（workaround、magic number）需注释原因

2. **核心逻辑必须添加测试用例**：
   - Happy Path：正常输入 → 预期输出
   - 边界值：空输入、最大/最小值、边界条件
   - 异常路径：无效输入 → 预期错误处理
   - 测试框架遵循项目现有约定（如 Jest、pytest、go test）

3. **遵循项目约束文档**：
   - 目录结构、命名规范、错误处理模式对齐现有代码
   - 如果修改涉及架构约束中声明的模块，需保持一致性

4. **搜索项目现有模式**：在编写代码前，先查看项目中类似功能的实现方式，保持风格一致。如果拿不准，可以搜索网络了解该语言/框架的最佳实践。

请完成此任务并报告：
1. 实际修改了哪些文件（与预估的差异）
2. 验收标准达成情况（逐条）
3. 添加了哪些测试用例（列出测试场景）
4. 已知限制或遗留问题
```

### 3.3 结果验证

每个子 agent 完成后，阅读其输出，逐项检查：

**验收维度：**
- [ ] 验收标准逐条达成
- [ ] 关键代码有注释（公共函数、复杂逻辑、非显而易见的决策）
- [ ] 核心逻辑有测试用例（Happy Path + 边界值 + 异常路径）
- [ ] 修改的文件与预估一致（如有偏差，评估是否合理）
- [ ] 代码风格与项目现有模式一致

**不通过处理：**
- 缺少注释 → 要求子 agent 补充后重新提交
- 缺少测试 → 如为核心逻辑，必须补测；非核心逻辑可记录为待补
- 风格不一致 → 指出差异，要求对齐

**通过后：**
- 如有遗留问题，记录到计划文档的"开放问题"中
- 验证通过的任务标记为 `[x]` 完成

### 3.4 失败处理

子 agent 执行失败时：
1. 分析失败原因（错误信息、不完整的输出）
2. 判断是否可重试：
   - **可重试**：网络问题、agent 超时 → 直接重试
   - **需调整**：任务描述不清晰、涉及范围超出预估 → 更新计划文档后再派发
   - **需人工介入**：需要业务决策、需要外部资源 → 暂停，告知用户
3. 更新计划文档中的风险记录

## 四、开发规范（子 Agent 必须遵守）

派发每个子 agent 时，必须在 prompt 中注入以下规范。

### 4.1 关键代码必须添加注释

| 场景 | 注释要求 | 示例 |
|------|---------|------|
| 公共函数/方法 | 说明用途、参数含义、返回值、副作用 | `// Authenticate user by phone/email. Returns session token, nil on failure.` |
| 复杂逻辑（>10 行或多分支） | 行内注释解释每一步的思路 | `// Step 1: normalize input to lowercase for case-insensitive match` |
| 非显而易见的决策 | 注释原因——为什么这样做而不是那样做 | `// Use LRU cache (not TTL) here because cache keys are long-lived` |
| Magic number / 硬编码值 | 注释含义和来源 | `// Timeout = 30s per API SLA (ref: docs/architecture.md#retry-policy)` |

**不需要注释的情况：**
- 自解释的 getter/setter
- 一眼能看懂的简单赋值
- 框架约定的标准模式（如 Express 中间件签名）

### 4.2 核心逻辑必须添加测试用例

**"核心逻辑"的判断标准：**
- 包含业务判断/分支逻辑的函数
- 数据处理/转换逻辑
- API 端点的请求处理
- 数据校验逻辑

**测试覆盖要求（至少 3 类场景）：**

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| Happy Path | 正常输入 → 预期输出 | 有效手机号+正确验证码 → 登录成功返回 token |
| 边界值 | 空输入、极值、边界条件 | 空字符串、超长输入、null/undefined、刚好达上限 |
| 异常路径 | 无效输入 → 预期错误 | 错误验证码 → 返回 401；过期 token → 返回 401 |

**测试框架选择：** 优先使用项目已有的测试框架和目录约定。如果项目无测试 → 使用语言标准框架（Jest、pytest、go test、JUnit）。

### 4.3 遵循项目约束文档

在派发前检查：
1. 项目是否有 `docs/architecture.md` 或 `CLAUDE.md` → 提取技术栈、目录结构、命名规范
2. 项目是否有 `.editorconfig`、`.eslintrc`、`pyproject.toml` 等 → 对齐代码风格
3. 项目已有类似模块 → 先 Read 理解模式，再动笔

**对齐检查清单：**
- [ ] 目录结构：新文件放在正确的位置
- [ ] 命名规范：文件名、函数名、变量名与项目一致
- [ ] 错误处理：使用项目统一的错误处理模式（异常/返回值/中间件）
- [ ] 日志风格：使用项目统一的日志格式和级别
- [ ] 依赖管理：不引入项目未使用的新依赖（除非任务明确要求）

### 4.4 搜索现有模式与最佳实践

**在编写代码前：**
1. 搜索项目中类似功能 → 理解现有实现模式
2. 搜索项目中的测试文件 → 理解测试约定（目录结构、命名、mock 方式）
3. 如果是新领域（如首次引入缓存），搜索网络了解该语言/框架的最佳实践

**搜索网络参考：**
- 语言/框架官方文档中的推荐模式
- 社区公认的最佳实践（如"Node.js error handling best practices"）
- 不直接复制粘贴——理解思路后用项目风格重写

## 五、Prompt 构建模板（汇总）

以下为派发子 agent 时使用的完整 prompt 模板，融合了任务信息 + 开发规范 + 约束文档：

```
## 任务：<SLUG>-PN-NN: <标题>

根据详细计划文档 <plan-path> 中的此任务。

### 任务描述
<从计划中复制>

### 验收标准
- [ ] <条件1>
- [ ] <条件2>

### 涉及文件（预估）
| 文件路径 | 操作 | 说明 |
|----------|------|------|
| <path> | 新增/修改 | <说明> |

### 前置依赖产出
<已完成任务的产出摘要，或 "None">

### 架构约束
<如果项目有约束文档，列出关键约束；否则写"遵循项目现有模式和约定">

### 开发规范（必须遵守）

1. **注释**：公共函数需说明用途/参数/返回值；复杂逻辑（>10行或多分支）需行内注释；magic number 需注释原因
2. **测试**：核心逻辑必须覆盖 Happy Path + 边界值 + 异常路径；使用项目现有测试框架
3. **约束**：遵循项目的目录结构、命名规范、错误处理模式
4. **风格**：先查看项目中类似实现，保持风格一致；不确定时可搜索网络最佳实践

### 完成后报告
1. 实际修改的文件（与预估的差异及原因）
2. 验收标准达成情况（逐条标记 [x]/[ ]）
3. 测试用例列表（场景描述 + 覆盖哪种情况）
4. 已知限制或遗留问题
```

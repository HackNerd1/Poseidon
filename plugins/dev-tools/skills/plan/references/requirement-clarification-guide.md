# 需求澄清指南

> 此文档为 plan skill 的 L3 参考文档。定义了需求澄清的六维评估框架、100 分制评分标准和提问模板。
> 在 SKILL.md Step 1 中按需加载。

---

## 一、六维评估框架

一个需求是否"明确"，可以从以下六个维度评估。每个维度有独立分值，总计 100 分。

### 维度一：目标清晰度（20 分）

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 问题描述清晰 | 7 | 要解决什么问题？为什么会成为问题？ |
| 预期成果明确 | 7 | "做完后是什么样的"——可感知的变化 |
| 验收标准可验证 | 6 | 有明确的 Given/When/Then 或 checkbox 清单 |

**不清晰的信号**：
- "把系统做好一点" → 什么叫"好"？
- "优化用户体验" → 具体优化哪些操作？怎么衡量？

### 维度二：范围边界（15 分）

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 范围内有清单 | 8 | 具体要做哪些事、涉及哪些模块/文件 |
| 范围外有清单 | 7 | 明确不做哪些事，防止范围蔓延 |

**不清晰的信号**：
- "顺便把那个也改一下" → 没有范围外清单
- "先做这些吧" → 隐含的后续范围未说明

### 维度三：功能明确性（30 分）

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 输入清晰 | 10 | 系统接收什么数据？格式？来源？ |
| 输出清晰 | 10 | 系统返回什么？格式？目的地？ |
| 交互方式确定 | 10 | 用户如何操作系统？CLI/API/UI？流程是什么？ |

**不清晰的信号**：
- "用户传个文件过来" → 什么格式？多大？从哪里传？
- "系统返回结果" → 什么结果？JSON？页面？通知？
- "大概是个表单" → 几个字段？校验规则？提交后发生什么？

### 维度四：技术约束（15 分）

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 技术栈明确 | 5 | 语言、框架、运行时、数据库 |
| 集成点已知 | 5 | 需要对接的外部 API、服务、中间件 |
| 约束条件清楚 | 5 | 性能指标、兼容性要求、安全要求、上线时间 |

**不清晰的信号**：
- "用最合适的技术" → 谁定义"合适"？
- "快一点" → 多少 ms 算快？P50 还是 P99？

### 维度五：业务上下文（20 分）

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 目标用户明确 | 7 | 谁会使用？什么角色？什么场景？ |
| 成功指标定义 | 7 | 做完后如何衡量成功？DAU？转化率？延迟？ |
| 优先级清晰 | 6 | 和其他需求相比的重要程度？是否阻塞其他工作？ |

**不清晰的信号**：
- "用户需要这个" → 哪些用户？多少个用户？
- "这个很重要" → 为什么重要？不做的后果是？

### 维度六：安全与回滚（加分项，不计入 100 分）

| 检查项 | 说明 |
|--------|------|
| 数据迁移方案 | 是否涉及已有数据的迁移？迁移失败怎么办？ |
| 回滚方案 | 上线后发现问题如何回退？回退窗口多长？ |
| 风险容忍度 | 可接受的最大影响范围？是否有 SLA 要求？ |

## 二、100 分评分制

### 评分流程

1. **首次评估**：基于用户提供的初始需求，按六维逐项打分，生成总分和雷达图信息
2. **增量更新**：每轮用户回答后，重新计算受影响维度的分数，告知变化（如"45→65，+20"）
3. **≥90 分**：进入计划阶段
4. **10 轮后仍 <90 分**：向用户说明情况，建议先进入计划，剩余不确定性在实施中迭代

### 分数含义

| 分数段 | 含义 | 建议 |
|--------|------|------|
| 0-30 | 严重缺失——仅有粗略想法 | 从目标和功能开始逐轮澄清 |
| 31-60 | 模糊——大致方向有，细节缺失 | 优先追问功能明确性和业务上下文 |
| 61-89 | 较清晰——多数维度有答案 | 追问剩余低分维度，可并行提问 |
| 90-100 | 清晰——可以进入计划阶段 | 确认理解后进入 Step 2 |

## 三、模糊词翻译表

需求描述中出现以下模糊词时，需追问将其量化为可观测指标：

| 模糊词 | 追问方向 |
|--------|---------|
| "快" / "高性能" | 响应时间目标是多少 ms？P50/P99？并发量？ |
| "用户友好" / "好用" | 具体的可用性指标？任务完成率？学习时间？ |
| "稳定" / "可靠" | 可用性 SLA？99.9%？允许的最大故障时间？ |
| "安全" | 认证方式？授权粒度？数据加密？审计日志？ |
| "可扩展" / "灵活" | 预期的增长规模？需要支持哪些扩展点？ |
| "支持" / "兼容" | 具体支持哪些版本/平台/格式？不支持的列表？ |
| "差不多" / "大概" | 精确的范围或阈值是多少？偏差容忍度？ |
| "几个" / "一些" | 具体数量？数量级（个/十/百/千/万）？ |
| "实时" | 延迟上限是多少 ms？准实时可接受吗？ |
| "无缝" | 用户感知到的切换/等待时间上限？ |

## 四、提问模板

### 开场白模板

```
在开始之前，我需要了解以下信息：
1. <问题1>（如果不关注，我会默认使用 <默认值>）
2. <问题2>
3. <问题3>
```

### 多选提问模板

```
关于 <主题>，你倾向于哪种方式？
A) <选项A> — <简要说明利弊>
B) <选项B> — <简要说明利弊>
C) <选项C> — <简要说明利弊>
如果拿不准，我推荐 <推荐选项>，因为 <原因>。
```

### 定义确认模板

```
你说需要"<模糊词>"，我理解为 <你的理解>。具体来说：
- 是否意味着 <可观测指标1> 达到 <阈值>？
- 是否意味着 <可观测指标2>？
请确认或修正。
```

### 验收标准引导模板

```
当这个功能完成时，你如何判断它"做对了"？
举几个例子：
- 用户做 <操作>，系统应该 <行为>
- 当 <异常情况> 发生时，系统应该 <处理方式>
能列 2-3 条这样的验收标准吗？
```

### 每轮结束模板

```
感谢你的回答。我的理解是：<1-2 句话总结>，对吗？

当前需求清晰度：<before> → <after>（<+N> 分）
剩余缺口主要在：<维度名称>

接下来我想了解：<下一轮问题预告>
```

## 五、反模式

| 反模式 | 为什么有问题 | 正确做法 |
|--------|-------------|---------|
| 一次性提 5+ 个问题 | 用户负担过重，回复质量下降 | 每轮 2-3 个精准问题 |
| 问可通过代码库回答的问题 | 浪费用户时间 | 先搜索项目文件自行确认 |
| 开放式问题过多 | 用户不知从何回答 | 给出选项或默认值 |
| 评分到 65 分就进入计划 | 关键维度可能缺失 | 坚持 ≥90 分或达到轮次上限 |
| 跳过确认直接生成计划 | 理解偏差可能导致返工 | 每轮结束时复述理解 |
| 忽视"范围外"维度 | 范围蔓延是项目最常见的失败原因 | 明确列出不做的事项 |

## 六、快速评分卡

评估时逐项打勾，快速计算分数：

```
□ 我知道要解决什么问题 (7)
□ 我知道做完后是什么样 (7)
□ 有可验证的验收标准 (6)
□ 有明确的范围内事项列表 (8)
□ 有明确的范围外事项列表 (7)
□ 输入的数据/操作已明确 (10)
□ 输出的结果已明确 (10)
□ 用户/系统的交互流程已确定 (10)
□ 技术栈已明确 (5)
□ 集成点已明确 (5)
□ 约束条件已明确 (5)
□ 目标用户已确定 (7)
□ 成功指标已定义 (7)
□ 优先级已明确 (6)
---
总分：__ / 100
```

## 七、澄清记录输出模板

澄清完成后，将结果写入 `docs/requirements/<feature-slug>-clarification.md`。格式如下：

```markdown
# 需求澄清记录 — <功能名称>

> 生成时间：<YYYY-MM-DD HH:MM>
> 最终评分：<score>/100
> 讨论轮次：<N> 轮

## 需求概述

<1-3 句话总结需求要解决的问题和预期成果>

## 澄清结果

### 目标与验收标准
- <条目1>
- <条目2>

### 功能描述
- **输入**：<描述>
- **输出**：<描述>
- **交互方式**：<描述>

### 范围边界
- **范围内**：<列表>
- **范围外**：<列表>

### 技术约束
- <条目1>

### 业务上下文
- **目标用户**：<描述>
- **成功指标**：<描述>

## 评分演进

| 轮次 | 评分 | 主要澄清内容 |
|------|------|-------------|
| 初始 | --/100 | — |
| 第1轮 | +N → **/100 | <澄清了哪些维度> |
| 第2轮 | +N → **/100 | <澄清了哪些维度> |
| ... | ... | ... |
| 最终 | **/100 | — |

## 未解决问题

<!-- 如果有超过 10 轮仍未解决或标记为"待后续迭代确认"的问题 -->
- <遗留问题1> — 计划在 Phase <N> 前确认
- <遗留问题2> — 需外部信息，暂缓

## 关键决策记录

<!-- 澄清过程中做出的重要决策 -->
- <决策1>：选择了 <方案A> 而非 <方案B>，因为 <原因>
```

### 使用说明

1. 此模板在 SKILL.md Step 1.5 中引用
2. 评分演进表记录每轮增量变化，展示澄清过程
3. 未解决问题不超过 3 条——超过说明澄清不充分
4. 关键决策记录帮助后续追溯"为什么这样选择"

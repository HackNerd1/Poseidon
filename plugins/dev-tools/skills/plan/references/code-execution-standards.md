# 代码实施规范

> 此文档为 plan skill 的 L3 参考文档，在 SKILL.md Step 3/4/5 中按需加载。
> 定义了派发子 agent 执行计划生成和代码实施时的开发规范、Prompt 模板和验收标准。

---

## 一、Agent 类型选择

| 场景 | Agent 类型 | 说明 |
|------|-----------|------|
| 生成总体计划（Step 3） | `Plan` | 软件架构 agent，擅长阶段划分、里程碑设计 |
| 生成详细计划（Step 4） | `Plan` | 软件架构 agent，擅长任务拆分、依赖分析 |
| 实施代码（Step 5） | `general-purpose` | 通用 agent，支持读写文件和执行命令 |

Plan agent 只负责生成计划文档，不写代码。general-purpose agent 负责实际代码实施。

## 二、Plan Agent Prompt 模板（Step 3 / Step 4）

### 2.1 总体计划生成（Step 3）

```
## 任务：生成总体计划 — <功能名称>

### 输入材料
- 需求文档：<path-to-requirement-doc>
- 澄清记录：<path-to-clarification-doc>
- 计划模板：references/overall-plan-template.md

### 任务要求
根据需求文档和澄清记录，按 overall-plan-template.md 的结构生成总体计划文档。

关键原则：
1. **垂直切片**：Phase 1 为最简端到端流程（Tracer Bullet），覆盖所有技术层但只实现 Happy Path
2. **阶段数量**：
   - 小型需求（<5 任务）→ 1 阶段
   - 中型需求（5-15 任务）→ 2-3 阶段
   - 大型需求（15+ 任务）→ 3-5 阶段
3. **依赖关系**：明确标注阶段间的依赖，允许并行的阶段标注为可并行

### 架构约束
<如果有项目约束文档，列出关键约束；否则写"基于通用最佳实践">

### 输出要求
1. 生成总体计划文档，写入 `docs/requirements/<feature-slug>-overall-plan.md`
2. 列出每个阶段的宏观任务（每阶段 2-5 个）
3. 标注阶段间的依赖关系
4. 为 Phase 1 标注为"待详细计划"

### 报告格式
完成后报告：
1. 计划文档路径
2. 阶段数量和各阶段任务数
3. 关键风险点
4. 建议的下一阶段（Phase 1）详细计划生成策略
```

### 2.2 详细计划生成（Step 4）

```
## 任务：生成详细计划 — <功能名称> — Phase <N>: <阶段名称>

### 输入材料
- 需求文档：<path-to-requirement-doc>
- 总体计划：<path-to-overall-plan>（如果有）
- 澄清记录：<path-to-clarification-doc>
- 计划模板：references/plan-template.md

### 任务要求
根据输入材料和 plan-template.md 的结构，为 Phase <N> 生成详细实施计划。

关键原则：
1. **垂直切片**：每个任务是一个完整的功能片段，而非水平技术层
2. **任务粒度**：每个任务 S(1-2文件) 或 M(3-5文件)，超过 5 文件需拆分，单阶段 3-7 个任务
3. **依赖清晰**：任务间依赖形成有向无环图，标注可并行任务
4. **可验证**：每个任务有明确的、二值的验收标准

### 架构约束
<如果项目有约束文档，列出关键约束>

### 输出要求
1. 生成详细计划文档，写入 `docs/requirements/<feature-slug>-detailed-plan-p<n>.md`
2. 任务 ID 格式：`<SLUG>-P<phase>-<seq>`（如 `LOGIN-P1-01`）
3. 包含依赖关系图（文本或 Mermaid）
4. 标注检查点（在哪些任务完成后验证什么）

### 报告格式
完成后报告：
1. 计划文档路径
2. 任务清单（ID + 标题 + 规模 + 依赖）
3. 可并行执行的任务组
4. 建议的派发策略（串行/并行/分批）
```

## 三、Code Agent Prompt 模板（Step 5）

### 3.1 完整 Prompt 模板

```
## 任务：<SLUG>-PN-NN: <标题>

根据详细计划文档 <plan-path> 中的此任务。

### 任务描述
<从计划中复制>

### 验收标准
- [ ] <条件1>
- [ ] <条件2>

### 涉及文件（预估）
| 文件路径 | 操作 | 说明 |
|----------|------|------|
| <path> | 新增/修改 | <说明> |

### 前置依赖产出
<已完成任务的产出摘要，或 "None">

### 架构约束
<如果项目有约束文档，列出关键约束>

### 开发规范（必须遵守）

1. **注释**：公共函数需说明用途/参数/返回值；复杂逻辑（>10行或多分支）需行内注释；magic number 需注释原因
2. **测试**：核心逻辑必须覆盖 Happy Path + 边界值 + 异常路径；使用项目现有测试框架
3. **约束**：遵循项目的目录结构、命名规范、错误处理模式
4. **风格**：先查看项目中类似实现，保持风格一致；不确定时可搜索网络最佳实践

### 完成后报告
1. 实际修改的文件（与预估的差异及原因）
2. 验收标准达成情况（逐条标记 [x]/[ ]）
3. 测试用例列表（场景描述 + 覆盖哪种情况）
4. 已知限制或遗留问题
```

### 3.2 并行派发时的补充说明

当一组无依赖任务并行派发时，在每个 agent prompt 末尾追加：

```
### 并行注意
此任务与以下任务并行执行：<列出其他并行的任务 ID>
请只修改你有权修改的文件，避免写入共享文件导致冲突。
```

## 四、开发规范详解

### 4.1 关键代码必须添加注释

| 场景 | 注释要求 | 示例 |
|------|---------|------|
| 公共函数/方法 | 说明用途、参数含义、返回值、副作用 | `// Authenticate user by phone/email. Returns session token, nil on failure.` |
| 复杂逻辑（>10 行或多分支） | 行内注释解释每一步的思路 | `// Step 1: normalize input to lowercase for case-insensitive match` |
| 非显而易见的决策 | 注释原因——为什么这样做而不是那样做 | `// Use LRU cache (not TTL) here because cache keys are long-lived` |
| Magic number / 硬编码值 | 注释含义和来源 | `// Timeout = 30s per API SLA (ref: docs/architecture.md#retry-policy)` |

**不需要注释的情况：** 自解释的 getter/setter、一眼能看懂的简单赋值、框架约定的标准模式。

### 4.2 核心逻辑必须添加测试用例

**"核心逻辑"的判断标准：**
- 包含业务判断/分支逻辑的函数
- 数据处理/转换逻辑
- API 端点的请求处理
- 数据校验逻辑

**测试覆盖要求（至少 3 类场景）：**

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| Happy Path | 正常输入 → 预期输出 | 有效手机号+正确验证码 → 登录成功返回 token |
| 边界值 | 空输入、极值、边界条件 | 空字符串、超长输入、null/undefined、刚好达上限 |
| 异常路径 | 无效输入 → 预期错误 | 错误验证码 → 返回 401；过期 token → 返回 401 |

**测试框架选择：** 优先使用项目已有的测试框架和目录约定。无现有测试基础设施 → 使用语言标准框架（Jest、pytest、go test、JUnit）。

### 4.3 遵循项目约束文档

**派发前检查：**
1. 项目是否有 `docs/architecture.md` 或 `CLAUDE.md` → 提取关键约束
2. 项目是否有 `.editorconfig`、`.eslintrc`、`pyproject.toml` 等 → 对齐代码风格
3. 项目已有类似模块 → 先 Read 理解模式，再动笔

**对齐检查清单：**
- [ ] 目录结构：新文件放在正确的位置
- [ ] 命名规范：文件名、函数名、变量名与项目一致
- [ ] 错误处理：使用项目统一的错误处理模式
- [ ] 日志风格：使用项目统一的日志格式和级别
- [ ] 依赖管理：不引入项目未使用的新依赖（除非任务明确要求）

### 4.4 搜索现有模式与最佳实践

**在编写代码前：**
1. 搜索项目中类似功能 → 理解现有实现模式
2. 搜索项目中的测试文件 → 理解测试约定（目录、命名、mock 方式）
3. 新领域（如首次引入缓存/消息队列）→ 搜索网络了解最佳实践

**搜索网络参考：** 语言/框架官方文档中的推荐模式 + 社区公认最佳实践。不直接复制粘贴——理解思路后用项目风格重写。

## 五、Agent 结果验收

### 5.1 Plan Agent 验收（Step 3/4）

- [ ] 计划文档已写入指定路径
- [ ] 文档结构符合对应模板（overall-plan-template / plan-template）
- [ ] 阶段/任务划分符合垂直切片原则
- [ ] 依赖关系明确，无循环依赖
- [ ] 每个任务有验收标准和预估文件列表
- [ ] 架构约束已纳入考量

### 5.2 Code Agent 验收（Step 5）

- [ ] 验收标准逐条达成
- [ ] 关键代码有注释（公共函数、复杂逻辑、非显而易见的决策）
- [ ] 核心逻辑有测试用例（Happy Path + 边界值 + 异常路径）
- [ ] 修改的文件与预估一致（如有偏差，评估是否合理）
- [ ] 代码风格与项目现有模式一致

### 5.3 不通过处理

| 问题 | 处理方式 |
|------|---------|
| 缺少注释 | 要求 agent 补充后重新提交 |
| 缺少测试（核心逻辑） | 必须补测 |
| 缺少测试（非核心） | 记录为待补，不阻塞流程 |
| 风格不一致 | 指出差异，要求对齐 |
| 验收标准未达成 | 分析原因，调整任务描述或重新派发 |

# 需求文档输出模板

> 此文档为 plan skill 的 L3 参考文档，在 SKILL.md Step 2 中按需加载。
> 定义了从澄清结果生成正式需求文档的标准输出格式。

---

## 文档结构

需求文档应包含以下章节。标注"（可选）"的可在不适用时省略。

```markdown
# 需求文档 — <功能名称>

> 版本：v1.0
> 创建时间：<YYYY-MM-DD HH:MM>
> 最后更新：<YYYY-MM-DD HH:MM>
> 关联澄清记录：`docs/requirements/<feature-slug>-clarification.md`

## 1. 背景与目标

- **问题描述**：<要解决的问题——为什么会成为问题？现状是什么？>
- **预期成果**：<要达到的目标——做完后"是什么样的"？>
- **目标用户**：<谁会使用？什么角色？什么场景下使用？>

## 2. 功能需求

### 2.1 核心功能
- <功能1：动词+成果，如"支持手机号+验证码登录">
- <功能2>
- <功能3>

### 2.2 输入/输出

- **输入**：<系统接收什么数据/操作——格式、来源、校验规则>
- **输出**：<系统返回什么结果——格式、内容、目的地>

### 2.3 交互流程

<用户与系统的交互步骤，可用文字描述或简单 ASCII 流程图>

```
用户 → 输入凭据 → 系统校验 → 成功/失败 → 跳转首页/提示错误
                                    → 记录登录日志
```

### 2.4 状态与异常（可选）

- **正常状态**：<Happy Path>
- **异常状态**：<至少列出 3 种异常情况及用户看到的内容>
  - <异常1>：<用户看到什么>
  - <异常2>：<用户看到什么>
  - <异常3>：<用户看到什么>

## 3. 非功能需求

- **性能**：<响应时间（P50/P99）、并发量、吞吐量>
- **安全**：<认证方式、授权粒度、数据加密、审计日志>
- **兼容性**：<浏览器版本、OS、运行时、依赖版本>
- **可用性**（可选）：<SLA 目标、允许的最大故障时间>

## 4. 验收标准

<!-- 可验证的、二值的（通过/不通过）条件 -->

- [ ] <验收条件1：Given/When/Then 格式>
- [ ] <验收条件2>
- [ ] <验收条件3>
- [ ] <验收条件4>

## 5. 范围边界

- **范围内**：
  - <事项1>
  - <事项2>
- **范围外**（明确不做）：
  - <事项1>
  - <事项2>

## 6. 约束与假设

- **技术约束**：<必须使用的技术栈、不能使用的技术>
- **业务假设**：<基于什么假设前提（如"用户已有账号"）>
- **时间约束**（可选）：<截止日期、里程碑日期>

## 7. 数据模型概述（可选）

<如果需求涉及数据结构变更，简述核心实体和关系>

## 8. 集成点（可选）

<如果需要对接外部系统/API，列出集成点和接口契约>
```

---

## 使用说明

此模板在 SKILL.md Step 2 中被加载。Claude 应：

1. 读取此模板了解需求文档的标准结构
2. 基于 Step 1 的澄清结果填充各章节
3. 对于不可知的字段标注"待确认"而非编造内容
4. 输出需求文档后告知用户路径，等待确认再进入 Step 3

## 质量检查清单

生成需求文档后自检：

- [ ] 背景与目标：用户看完知道"为什么做"
- [ ] 功能需求：开发者看完知道"做什么"
- [ ] 输入/输出：数据格式和交互流程明确
- [ ] 非功能需求：性能/安全/兼容性有具体指标（非"快""安全"等模糊词）
- [ ] 验收标准：每条都可独立验证（二值：通过/不通过）
- [ ] 范围边界：范围外有明确清单（防止范围蔓延）
- [ ] 关联澄清记录：可追溯需求来源

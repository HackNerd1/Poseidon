# Maintenance Guide

## 用途

本指南用于维护 `job-opportunity-analyzer` 的事实源资产，覆盖公司池样例、来源清单、报告样例和字段规则的更新方式。它服务于人工维护和后续脚本实现，不负责开放式分析。

## 维护对象

- `assets/company-pool.schema.json`
  - 结构契约来源，任何字段变更都要先更新这里。
- `assets/<city>-company-pool.sample.json`
  - 城市级样例事实源，用于验证字段布局和报告输入。
- `references/company-discovery-guide.md`
  - 企业发现方法和来源优先级。
- `references/opportunity-source-guide.md`
  - 岗位采集与风评来源约束。
- `references/report-output-contract.md`
  - 最终报告的输出契约。
- `references/data-model.md`
  - 字段含义、版本规则和样例命名说明。

## 更新触发条件

出现以下情况时应更新资料：

- 新增城市样例，例如从杭州扩展到上海或深圳。
- 需要新增可选字段，但不改变现有语义。
- 发现已有来源长期失效，无法继续支持企业发现或岗位验证。
- 报告输出需要新增固定章节或异常说明。
- 主 skill 引入新的过滤条件，现有 schema 或参考文档无法解释。

## 推荐维护节奏

首期采用轻维护、人工复核优先的节奏。以下频率是建议值，不是实时 SLA：

| 维护对象 | 建议频率 | 原因 | 触发人工复核的典型情况 |
|----------|----------|------|------------------------|
| 榜单 / 创投 / 独角兽 / 高成长名单 | 每季度一次 | 这类来源更新慢，适合用来扩充候选池，不值得按周追 | 排名口径明显变化、榜单下线、同一公司连续两期消失 |
| 地方重点企业名单 / 拟上市名单 / 产业链名单 | 每月一次 | 地方名单用于发现本地机会，政策和招商口径会调整 | 名单链接失效、政府页面迁移、企业主营方向变化明显 |
| 高校就业网 / 招聘会公告 | 求职季每月一次，淡季按季度 | 季节性强，招聘活跃度波动大，需要按招聘周期观察 | 长时间只剩过期岗位、招聘会改版、公告不再覆盖目标岗位 |
| 招聘平台职位页 / 公司招聘页 | 每次人工分析时校验，活跃样例建议每月复看一次 | 岗位时效性最强，但首期不做自动监控，只在使用时复核 | 职位下线、薪资或地点冲突、职位链接跳转异常 |
| 社区 / 新闻 / 公开风评信号 | 每次生成报告时抽样复核，常用样例建议每季度复看一次 | 风评是辅助信号，容易受时间窗口和样本偏差影响 | 样本过少、信号互相冲突、内容疑似断章取义 |
| schema / data-model / 输出契约 | 仅在需求或输出结构变化时更新 | 结构文档不应为了“刷新”而刷新，避免无意义 churn | 主 skill 输入输出新增字段、样例已无法表达真实结论 |
| 城市样例与报告样例 | 有新增城市 / 岗位方向时更新；既有样例建议每季度抽查 | 样例是演示和回归基线，不要求实时，但要避免长期过期 | 样例中多家企业已失效、报告引用字段与当前规则不一致 |

## 维护流程

### 1. 先判断变更类型

- 结构变更
  - 影响 schema、字段解释或脚本输入输出。
- 来源变更
  - 影响企业发现、岗位采集、风评整理的参考资料。
- 样例变更
  - 影响城市样例内容，但不改字段结构。

### 1.1 再判断维护动作

- 新增
  - 新增城市、岗位方向、来源类别或候选公司。
- 更新
  - 现有来源仍有效，但字段、岗位事实、风评摘要需要刷新。
- 下线
  - 来源、公司或岗位不再适合作为候选事实源，需要降级、替换或移出活跃样例。
- 复核
  - 信息存在冲突、样本不足或业务偏移，脚本无法自动裁决时人工确认。

### 2. 再决定落点

- 结构语义变化
  - 更新 `assets/company-pool.schema.json`
  - 同步更新 `references/data-model.md`
- 来源方法变化
  - 更新对应 guide 文档
- 样例内容变化
  - 更新 `assets/<city>-company-pool.sample.json`
  - 如有字段新增，再回到结构语义流程

### 2.1 来源失效与替换规则

- 链接短时失效
  - 首次发现 404、重定向异常或权限错误时，先保留原记录并标记“待复核”，不要立刻删除。
  - 下次维护仍失效时，优先替换为同一来源类别下的等价公开页面，例如官网职位页替换旧职位直链、政府名单新地址替换旧地址。
- 来源长期失效
  - 连续两次维护窗口都无法访问，且无法找到同类替代页面时，将该来源从活跃推荐列表降级。
  - 降级后可以保留在历史样例里作为注释性证据，但不能继续作为“当前可投机会”的主要依据。
- 只有第三方来源
  - 若官网缺失而第三方职位页仍在，允许保留公司候选，但要显式标记 `third_party_only`，并触发人工复核。
- 风评来源失效
  - 单条社区或新闻链接失效时，不自动删除整段风评结论；先检查是否还有其他证据支撑。
  - 如果关键正负信号已无可追溯证据，降级为“样本不足”，并要求人工重写摘要或移除结论。

### 2.2 公司池清理与降级规则

- 公司已倒闭、注销或公开确认停止相关业务
  - 从活跃候选池移出，不再参与机会对比。
  - 如该公司对历史样例有说明价值，可保留在历史注释中，但必须写明失效原因和观察时间。
- 长期无招聘
  - 连续两个维护周期都没有目标岗位方向的公开招聘信号时，从“优先跟进”降级为“观察名单”。
  - 如果连续三个维护周期仍无相关岗位，且无新业务扩张信号，可从城市样例中移出。
- 业务明显偏离目标岗位
  - 例如公司主营已转向与目标岗位弱相关的外包、线下业务或非技术导向场景，应从主候选池降级。
  - 降级后如仍保留，必须在 `match_reason` 或备注中说明偏离原因，避免误导后续分析。
- 字段长期缺失
  - 多次维护后仍拿不到基本岗位事实，例如工作地、经验要求或职位链接，应停止把该机会作为可比较样例。

### 3. 最后做确定性检查

每次修改结构化文件后，至少运行一次本 skill 的校验脚本：

```bash
python3 plugins/job/skills/job-opportunity-analyzer/scripts/validate-company-pool.py
```

如果是指定城市样例，也可以显式传参：

```bash
python3 plugins/job/skills/job-opportunity-analyzer/scripts/validate-company-pool.py \
  --sample plugins/job/skills/job-opportunity-analyzer/assets/hangzhou-company-pool.sample.json
```

## 人工复核要求

以下内容不能只靠脚本，必须人工确认：

- 该公司是否真的与目标岗位方向相关
- 社区或新闻风评是否被断章取义
- 来源是否足以支撑“优先投递”这类判断
- 城市覆盖度是否已经足够，不再需要补公司
- 链接失效后是否存在同等级替代来源，而不是误把暂时故障当作永久下线
- 公司是否只是短期停止招聘，还是已经不适合作为目标候选
- 岗位信息是否因为节假日、招聘季切换或页面改版而显得陈旧

### 触发人工复核的条件

- 同一岗位的薪资、城市、年限要求出现明显冲突
- 官网信息缺失，只能依赖第三方职位页或二手转载
- 风评样本少于两类独立来源，或正负信号高度冲突
- 公司名称仍在来源名单中，但公开业务、融资、招聘主题已经明显偏离目标岗位
- 同一来源连续两次失效，准备执行替换或下线动作前
- 计划把公司从活跃池移出，或把机会从“可投”降级为“待人工判断”前

原因：

- 这些判断依赖上下文、业务理解和证据权重，不属于确定性校验。

## Scripts 应承担的职责

- 样例文件是否存在且能被 JSON 解析
- 顶层字段是否齐全
- `company_id`、`opportunity_id`、`sentiment_id` 是否满足唯一性要求
- `job_opportunities` / `company_sentiments` 的 `company_id` 是否能回指到已定义公司
- `source_refs` 是否存在且非空

## Scripts 不应承担的职责

- 自动发现本地公司
- 生成岗位匹配结论
- 判断风评可信度
- 对冲突来源做取舍裁决
- 实现真实抓取、登录或反爬逻辑

## 异常处理约定

- 来源暂时失效
  - 保留来源类别并在报告中写明失效，不要为了“通过校验”删除事实。
- 来源需要替换
  - 优先替换为同类别、同权重的公开来源；替换后同步检查 `source_id`、标题和观察时间是否仍一致。
- 公司状态变化
  - 已倒闭、注销或停止相关业务时，从活跃候选池移出；如保留历史记录，必须注明“历史样例”。
- 岗位信息陈旧
  - 超过一个建议维护周期仍未刷新时，不继续把它写成当前机会，优先降级为观察项或移出报告对比表。
- 样例字段不足
  - 先补齐样例，再决定是否需要调整 schema。
- 发现字段语义不清
  - 先更新 `data-model.md`，再决定是否修改样例和脚本。

## 非目标

- 不定义跨城市汇总库的运营流程
- 不规定定时任务或自动刷新机制
- 不替代后续 Phase 3 的真实校验脚本实现

# Opportunity And Sentiment Source Guide

## 用途

本指南用于 `SKILL.md` 的 Step 3 和 Step 4，帮助整理岗位事实、薪资文本、经验要求和公司风评来源。

## 目标

- 为每个候选公司补齐岗位和风评证据
- 区分“原始事实”和“分析判断”
- 在信息不完整时保留异常说明

## 推荐来源类型

### 岗位来源

1. **官方招聘页 / 官方招聘公众号 / 官方校招页**
   - 用于确认公司是否真的在招、岗位名称是否存在、工作地和团队描述是否明确
   - 默认可信度：`high`
2. **公开招聘平台职位页**
   - 用于补充薪资、年限、团队方向、标签、岗位活跃度
   - 默认可信度：`medium`
3. **高校就业网 / 校招聚合页**
   - 用于发现实习、校招或特定技术栈机会，也可补充宣讲城市与届别信息
   - 默认可信度：`medium`

### 风评来源

1. **公开新闻与公司动态**
   - 用于判断融资、裁员、业务扩张、产品方向变化
   - 默认可信度：`medium-high`
2. **技术社区 / 求职社区讨论**
   - 用于观察团队口碑、工作节奏、管理风险
   - 默认可信度：`low-medium`
3. **公开问答或经验贴**
   - 用于补充面试体验、薪资风格、组织稳定性信号
   - 默认可信度：`low-medium`

## 来源优先级

### 岗位事实优先级

| 优先级 | 来源类型 | 主要用途 | 可直接作为主事实 | 使用规则 |
|--------|----------|----------|------------------|---------|
| P0 | 官网 / 官方招聘系统 / 官方校招页 | 确认岗位存在、岗位名、工作地、团队描述、投递入口 | 是 | 只要能获取到官方版本，就把它作为岗位主记录 |
| P1 | 招聘平台职位页 | 补薪资、经验要求、技术栈标签、招聘活跃度 | 条件式 | 当 P0 缺字段时可补充；若和 P0 冲突，必须保留冲突并标记 |
| P2 | 高校就业网 / 宣讲页 / 聚合页 | 补届别、宣讲城市、校招批次 | 条件式 | 仅用于校招/实习补充，不覆盖 P0/P1 的通用岗位字段 |
| P3 | 社区转帖 / 转载汇总 | 提供线索 | 否 | 只能作为待验证线索，不进入主事实字段 |

规则：
- 官网优先，第三方补充，社区转帖只做线索，不单独生成“确定岗位事实”。
- 同一岗位优先保留能直接投递或能定位到官方页面的版本。
- 若第三方平台有薪资或年限而官网没有，可以补入结构化结果，但必须保留来源引用和观察时间。

### 风评信号优先级

| 优先级 | 来源类型 | 主要用途 | 使用规则 |
|--------|----------|----------|---------|
| S1 | 官方新闻稿 / 财报 / 官方博客 | 判断业务扩张、产品方向、组织公开表态 | 可作背景事实，但不直接代表员工体验 |
| S2 | 主流新闻 / 行业媒体 | 补充融资、裁员、诉讼、监管、业务波动 | 可作风险背景，需记录发布时间 |
| S3 | 面经 / 社区讨论 / 匿名分享 | 观察工作节奏、面试流程、团队口碑 | 只能作辅助信号，不能单点定性 |
| S4 | 无法定位出处的截图或二手转述 | 线索提示 | 默认不采纳为正式风评事实 |

规则：
- 新闻和官方动态用于“发生了什么”，社区讨论用于“可能意味着什么”。
- 社区和新闻只能作辅助信号，不能替代岗位事实，也不能单独决定推荐/否定。
- 风评摘要至少要带时间和证据引用，避免把过期讨论当成当前状态。

## 整理规则

### 原始事实

每条事实建议至少保留：

- 来源类型
- 链接或可定位页面
- 观察时间
- 10-40 字摘要
- 原文摘录或关键原词

### 分析判断

在原始事实之后，再给出：

- `match_reason`：为什么与用户岗位方向匹配
- `risk_flags`：有哪些风险信号
- `confidence`：高 / 中 / 低

原因说明：
- 风评类来源主观性强，因此必须和岗位或官方事实分开记录，避免“传闻即结论”。

## 字段对齐规则

### 岗位字段映射

| 目标字段 | 主要来源 | 保留规则 | 允许的轻量归一化 | 冲突处理 |
|----------|----------|----------|------------------|---------|
| `job_title` | 官网、招聘平台、高校就业网 | 保留原始岗位名，避免擅自改写层级词如“高级/资深/专家” | 可附 `normalized_job_family`，如 `frontend-engineer` | 若多个来源标题不同，保留官方标题为主，第三方标题进入冲突说明 |
| `tech_stack` | 官网、招聘平台 | 保留原文技术关键词，便于后续匹配 | 可去重并拆分成关键词列表，如 React / Vue / TypeScript | 平台补充但官网未写时，标为“third_party_only” |
| `experience_requirement` | 官网、招聘平台 | 保留原文，如 `3-5年`、`资深优先` | 仅在文本明确时映射到区间标签，如 `0-1`、`3-5` | 若官网和平台冲突，保留两者并标记 `experience_unaligned` |
| `salary_text` | 招聘平台、校招页、社区面经 | 必须保留薪资原文，不能只留归一化数字 | 可补一个粗粒度说明，如“月薪区间”或“总包不明确” | 多来源冲突时保留全部原文并标记 `salary_unaligned` |
| `location` | 官网、招聘平台、高校就业网 | 保留原文城市/区县/办公点 | 可抽取城市主标签，如 `hangzhou` | 出现异地/多地/混合办公冲突时并列展示并标记 |
| `job_url` | 官网、招聘平台 | 保留可访问的直接链接 | 不做归一化 | 若官方和平台都可投，优先官方链接作为主链接 |
| `source_type` | 全部 | 记录事实来自哪类来源 | 不需要 | 不冲突，允许多值 |
| `team_description` | 官网、招聘平台、新闻 | 保留原文摘要，不改写业务范围 | 可压缩成 10-30 字摘要 | 描述冲突时不裁决，进入 `signal_conflict` |

说明：
- 岗位名、薪资原文、经验要求、团队描述优先保留原文，因为这些字段最容易在不同来源间出现语义漂移。
- 轻量归一化只服务于筛选和比较，不能替代原文，也不能抹平来源差异。

### 风评字段映射

| 目标字段 | 主要来源 | 保留规则 | 允许的轻量归一化 | 冲突处理 |
|----------|----------|----------|------------------|---------|
| `sentiment_source_category` | 新闻、社区、问答、官方动态 | 必填，必须指明类别 | 可映射为 `official` / `news` / `community` / `qa` | 不冲突 |
| `positive_signals` | 新闻、社区、官方动态 | 每条信号单独记录，不做大段总结替代证据 | 可归类为业务增长、技术投入、招聘扩张等标签 | 与负向信号并存时，不相互抵消 |
| `negative_signals` | 新闻、社区、问答 | 每条信号单独记录，并附时间 | 可归类为裁员、加班、流程混乱、HC 收缩等标签 | 若样本稀少，标记 `insufficient_sentiment_samples` |
| `sentiment_summary` | 综合整理 | 只能在保留证据后写 1-3 句摘要 | 可分成“支持点 / 保留点” | 存在显著冲突时标记 `signal_conflict` |
| `observation_time` | 全部 | 必填，至少精确到日期或月份 | 可统一成 `YYYY-MM` 或 `YYYY-MM-DD` | 时间跨度过大时需提示“可能过期” |
| `evidence_refs` | 全部 | 必填，至少 1 条可定位证据 | 可用编号引用 | 若引用失效，保留失效说明，不删除整条结论 |
| `confidence` | 综合整理 | 只允许 `high` / `medium` / `low` | 依据来源层级和样本数量判定 | 冲突越大、样本越少，置信度越低 |

说明：
- 风评摘要必须晚于证据整理出现，不能先下结论再反推证据。
- 正向和负向信号允许同时存在，避免把复杂情况压成单一“好/坏”结论。

### 薪资

- 若来源给出明确区间，保留原文本并可附一个轻量标准化解释
- 若出现 `面议`、`薪资保密`、`根据能力定级`，直接标为 `salary_unaligned`
- 若不同来源冲突，不做强行合并，保留多个事实

### 经验要求

- `1-3年`、`3年以上`、`资深优先` 这类写法全部先保留原文
- 只有当文本可以明确映射到用户过滤条件时，才做通过/不通过判断
- 无法对齐时标记 `experience_unaligned`

### 工作地

- `杭州`、`杭州/上海`、`支持异地`、`滨江/余杭` 这类写法都先保留原文
- 允许补一个城市主标签，便于和用户输入位置匹配
- 若官网写 `杭州` 而平台写 `杭州/上海`，不裁决哪个“更真”，直接标记 `location_unaligned`

### 团队描述

- 官网团队描述优先作为背景事实
- 招聘平台或新闻中出现不同业务线名称时，保留原文并标记 `signal_conflict`
- 不根据单条社区帖子推断“团队稳定/不稳定”

## 风险标签与状态标记

| 标签 | 触发条件 | 处理方式 |
|------|----------|---------|
| `salary_unaligned` | 薪资原文冲突、薪资保密、或无法映射到用户期望 | 保留全部薪资原文，并提示人工复核 |
| `experience_unaligned` | 年限写法冲突或语义过于模糊 | 保留原文，不做强过滤 |
| `location_unaligned` | 工作地、办公模式、城市范围不一致 | 并列展示来源差异 |
| `signal_conflict` | 团队描述、业务状态、风评正负信号相互矛盾 | 输出摘要时显式写出分歧点 |
| `insufficient_sentiment_samples` | 风评来源少于 2 条独立样本，或只有单一社区来源 | 降低 `confidence`，提示仅供参考 |
| `third_party_only` | 关键字段只在第三方平台出现，官网缺失 | 标记为待官方补证，不当作完全确认事实 |

规则：
- 风险标签是“提醒人工复核”的机制，不是“自动判定真伪”的机制。
- 只要存在关键字段冲突，就保留冲突事实，不输出“已自动修正”的说法。

## 冲突处理流程

1. 先保留各来源原文和观察时间，不提前合并。
2. 用来源优先级确定“主展示字段”。
3. 只对可机械归类的部分做轻量归一化，如技术栈去重、城市主标签抽取。
4. 一旦岗位薪资、年限、工作地、团队描述出现冲突：
   - 保留主来源值
   - 保留冲突来源值
   - 添加对应风险标签
   - 在摘要中提示“需人工复核”
5. 不承诺自动判定哪条一定正确，尤其是社区与新闻、平台与官网冲突时。

## 风评使用约束

- 风评只能作为风险提示或补充信息，不能单独作为推荐/否定公司的唯一依据
- 同一结论尽量至少有两个独立来源，或与新闻/官方动态互相印证
- 如果只有单条社区帖子，不要写成确定事实

## 异常处理

- **来源失效**：记录来源类别和失效情况，继续使用剩余来源
- **内容冲突**：并列展示冲突事实，不擅自裁决
- **岗位描述极少**：保留公司层级结论，但降低岗位匹配置信度
- **风评样本不足**：明确写出“风评样本不足”

## 不要做的事

- 不要承诺实时最新职位数据
- 不要把个人主观看法写成事实源
- 不要在本指南里塞入报告模板；报告结构由单独的输出契约管理

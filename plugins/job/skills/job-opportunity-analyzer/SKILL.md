# Job Opportunity Analyzer

## 概述

你将围绕目标城市、岗位关键词和候选公司池整理公开求职信息，输出一份可追溯的机会分析结果。核心原则：**先限定范围，再补事实源，最后输出带引用和缺口说明的 Markdown 报告**。

该 skill 采用半自动方案：
- 优先使用本地事实源、已有公司池和公开可访问来源
- 允许搜索补充信息，但不承诺实时抓取、批量爬取或登录态访问
- 报告必须能说明结论来自哪里，以及哪些点仍需人工复核

## 输入要求

### 必填输入

- `location`：目标城市或城市群，如 `hangzhou`、`shanghai`
- `position`：岗位关键词，支持职位名或方向词，如 `frontend`、`ai product manager`

### 可选输入

- `salary_min`：期望最低月薪；未提供则不做最低薪资过滤
- `salary_max`：期望最高月薪；未提供则不做最高薪资过滤
- `experience_min`：最低可接受经验要求；未提供则默认按 `0` 年处理
- `experience_max`：最高可接受经验要求；未提供则不做上限过滤
- `optional_companies`：用户指定或偏好的公司名单；提供时优先纳入候选池，而不是替代城市发现流程
- `must_include_sources`：用户要求必须覆盖的来源类别；未提供则按默认来源清单执行
- `report_focus`：用户更关注的输出重点，可为 `salary`、`stability`、`growth`、`brand`；未提供则默认做平衡分析

### 默认处理规则

- 城市参数优先使用用户原文；只有在明显是英文 slug 时才按 slug 使用，不强制改写用户表达，避免地名误归一。
- 岗位关键词默认做同义词扩展，例如 `frontend` 可扩展到 `web`、`h5`、`react`，但必须在报告中标记哪些词是推断补充。
- 薪资和经验要求仅在来源能直接映射时参与过滤；无法直接对齐时保留原始文本并标记 `需人工复核`，避免假精确。
- 用户未给 `optional_companies` 时，仍需先尝试城市发现，而不是直接输出“无公司可分析”。

## 核心流程

### Step 1: 校验输入并确认分析范围

1. 检查 `location` 和 `position` 是否存在；缺少任一项时先向用户补齐。
2. 整理分析约束：
   - 城市
   - 岗位关键词
   - 薪资/经验过滤条件
   - 用户指定公司
   - 报告重点
3. 向用户复述本次分析范围，避免把“找公司”和“评估已给公司名单”混成一个任务。

### Step 2: 构建候选公司池

按需读取 `references/company-discovery-guide.md`。

1. 先检查是否已有适用的城市公司池或候选名单。
2. 若无预置公司池，按 reference 中的来源分层补齐：
   - 招聘平台公开职位页
   - 高校就业网 / 校企合作名录
   - 地方产业名单、拟上市名单、创投榜单
   - 用户给定公司和已知行业名单
3. 将发现到的公司按“为什么进入候选池”做简要归因，避免只给名单不给理由。
4. 如果城市覆盖不足，明确记录“当前城市无预置公司池，已转为动态发现模式”。

### Step 3: 整理岗位、薪资和公司风评事实

按需读取 `references/opportunity-source-guide.md`。

1. 针对每家候选公司，优先整理：
   - 官方招聘页或公司官网说明
   - 公开岗位页中的岗位名称、经验要求、薪资文本、地点
   - 社区风评、公开新闻或讨论中的正负面事实
2. 岗位与风评必须区分“原始事实”和“分析结论”：
   - 原始事实：链接、原文摘要、观察时间
   - 分析结论：匹配原因、风险提示、待验证项
3. 若同一公司的多个来源互相冲突，保留冲突并说明，不自行拍板哪个一定正确。

### Step 4: 按条件筛选并形成机会判断

1. 用岗位关键词、薪资、经验要求筛掉明显不匹配的机会。
2. 对无法结构化对齐的字段按以下方式处理：
   - 薪资描述模糊：保留原始文本，标记 `salary_unaligned`
   - 年限写法不统一：保留原始文本，标记 `experience_unaligned`
   - 来源只描述团队方向、不含岗位细节：保留公司但下调机会置信度
3. 输出每个保留机会的简要结论：
   - 为什么值得看
   - 关键风险是什么
   - 还缺什么信息

### Step 5: 生成报告与异常说明

按需读取 `references/report-output-contract.md`。

1. 输出 Markdown 报告。
2. 报告中必须同时包含：
   - 结论摘要
   - 候选公司/岗位机会表
   - 事实源引用
   - 缺失项和异常说明
   - 后续人工核查建议
3. 如果来源严重不足，也要输出“可执行但不完整”的报告，而不是空结果。

## 输出格式

默认输出单份 Markdown 报告，结构至少包括：

```markdown
# Job Opportunity Report

## 1. Analysis Scope
- location:
- position:
- filters:

## 2. Executive Summary
- 推荐关注的公司/机会
- 主要判断依据

## 3. Opportunity Table
| company | role | salary | experience | match_reason | risk_flags | source_refs |

## 4. Company Notes
- 每家公司 2-5 条事实与判断

## 5. Fact Sources
- [Source ID] 来源类型 | 链接/文件 | 观察时间 | 用途

## 6. Missing Data And Exceptions
- 城市无预置公司池 / 来源失效 / 薪资无法对齐 / 年限无法对齐 / 风评样本不足

## 7. Suggested Next Checks
- 建议人工复核的页面、字段或公司
```

输出要求：
- 必须有事实源引用，不能只有结论
- 必须写明异常或缺口，不能把未知信息伪装成确定结论
- 如使用了推断扩展词或人工归类，需要在报告中显式注明

## 质量规则

1. **事实先于判断**：每个核心结论至少对应一个可追溯来源。
2. **来源分层**：官方来源优先于招聘平台，招聘平台优先于社区讨论；社区风评只能作为风险信号，不能单独充当结论依据。
3. **缺口显式化**：来源不足、文本不一致、字段无法结构化时必须写进异常说明。
4. **半自动边界**：可以搜索公开信息，但不要承诺“全网完整覆盖”或“实时最新”。
5. **保留原文**：薪资、年限、风评等关键信息无法标准化时，优先保留原始描述。
6. **范围稳定**：首期只做求职机会分析，不延伸到简历改写、自动投递、定时跟踪。

## 边界情况

- **城市无预置公司池**：转入动态发现模式，按 `company-discovery-guide.md` 的来源类别补齐，并在报告中标记当前覆盖度有限。
- **来源失效或无法访问**：跳过该来源，记录失效链接和受影响字段，提示用户后续人工补查。
- **薪资无法直接对齐**：保留原始薪资文本，例如 `20-30K·14薪`、`面议`，标记 `需人工复核`。
- **年限无法直接对齐**：保留原始要求，例如 `3年以上`、`资深优先`，标记 `需人工复核`。
- **公司风评互相矛盾**：保留正反两侧事实，说明样本来源和时间，不强行合并为单一结论。
- **用户只给公司名单、不关心城市发现**：可缩短 Step 2，但仍需检查这些公司是否覆盖用户岗位方向。
- **公开来源信息极少**：输出最小可用报告，并把“需要用户补充的公司或来源”列为下一步动作。

## 限制

- 不处理登录站点、复杂反爬、批量抓取和后台定时刷新。
- 不承诺产出实时、完备或可直接投递的职位数据库。
- 不在本任务中定义 schema、样例数据或脚本实现细节；这些内容应在后续任务中补齐。
- 该 skill 适合机会分析和来源整理，不适合自动化投递、消息通知或简历重写。

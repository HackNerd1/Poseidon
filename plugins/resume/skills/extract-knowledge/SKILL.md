# 提取项目知识工作流

## 概述

从用户提供的 Git 仓库（远程 URL 或本地路径）中，深入分析代码结构、文档和配置，挖掘三个维度的核心信息：

1. **业务场景**：核心业务流程、领域模型、业务规则
2. **技术 & 难点**：技术选型、架构模式、性能优化、安全策略、技术难点攻克
3. **关键设计**：模块划分、设计模式、关键抽象、扩展机制、权衡决策

按维度组织输出结构化知识文档，保存到 `project-info/<project_name>.md` 或用户指定目录。

## 工作流

### 第一步：解析输入

从用户提供的参数中判断：

1. **远程 URL**（含 `github.com` / `gitlab.com` / `gitee.com` 等）→ 进入第二步 Clone
2. **本地路径**（如 `/path/to/project` 或 `E:\project\xxx`）→ 直接使用，跳到第三步
3. **用户未提供** → 询问用户提供仓库地址

提取项目名称：
- URL：从路径末段提取，去掉 `.git` 后缀
- 本地路径：使用目录名

确认输出路径：
- 用户指定了第二个参数 → 使用用户指定的路径
- 未指定 → 默认 `project-info/<project_name>.md`

向用户确认项目名称和输出路径。

### 第二步：Clone 仓库

将远程仓库 clone 到 `.Poseidon/repo/<project_name>/`：

```bash
git clone <repo_url> ".Poseidon/repo/<project_name>/"
```

如果 `.Poseidon/repo/<project_name>/` 已存在，询问用户是否复用（跳过 clone）还是重新 clone。

### 第三步：初步扫描

快速扫描项目，向用户报告基本信息后确认进入分析：

```
📋 项目扫描结果

**项目名称**：<project_name>
**项目类型**：<Web 应用 / CLI 工具 / 库/SDK / 移动端 / 桌面端 / 其他>
**主要语言**：<按占比排序，前 3 项>
**核心框架**：<框架名>
**目录规模**：<文件数> 文件，<目录数> 目录
**预估分析范围**：<小 / 中 / 大>

确认开始深入分析？
```

扫描方法：
- 读取 README.md（如有）了解项目定位
- 读取 package.json / go.mod / Cargo.toml / requirements.txt 等依赖文件获取技术栈
- 遍历顶层目录结构了解模块划分

### 第四步：并行派发子 Agent 分析

**同时派发三个子 Agent**，每个读取对应的 reference worker contract：

| 子 Agent | Reference（Worker Contract） | 输出文件 |
|---------|---------------------------|----------|
| Agent A | `references/business-scenario-contract.md` | `.Poseidon/repo/<project_name>/analysis/business-scenarios.md` |
| Agent B | `references/tech-challenge-contract.md` | `.Poseidon/repo/<project_name>/analysis/tech-challenges.md` |
| Agent C | `references/design-decision-contract.md` | `.Poseidon/repo/<project_name>/analysis/design-decisions.md` |

创建输出目录：

```bash
mkdir -p ".Poseidon/repo/<project_name>/analysis"
```

派发方式：

```
Agent(description: "业务场景分析", prompt: "读取 references/business-scenario-contract.md 的完整契约，对 .Poseidon/repo/<project_name>/ 执行业务场景分析，结果写入 .Poseidon/repo/<project_name>/analysis/business-scenarios.md")
Agent(description: "技术架构与难点分析", prompt: "读取 references/tech-challenge-contract.md 的完整契约，对 .Poseidon/repo/<project_name>/ 执行技术架构与难点分析，结果写入 .Poseidon/repo/<project_name>/analysis/tech-challenges.md")
Agent(description: "设计决策分析", prompt: "读取 references/design-decision-contract.md 的完整契约，对 .Poseidon/repo/<project_name>/ 执行关键设计决策分析，结果写入 .Poseidon/repo/<project_name>/analysis/design-decisions.md")
```

**等待所有三个子 Agent 完成后**，进入第五步。

### 第五步：结果校验

逐个读取三个中间文件，检查质量：

**business-scenarios.md 检查：**
- 是否识别了 2-5 个核心业务场景？
- 每个场景是否有清晰的流程描述？
- 是否提取了领域模型和业务规则？

**tech-challenges.md 检查：**
- 技术栈是否按核心/辅助/基础设施三层分类？
- 是否识别了架构模式？
- 技术难点是否具体（有实际问题背景和解决方案，而非罗列技术名词）？

**design-decisions.md 检查：**
- 模块划分是否清晰？
- 是否识别了具体的设计模式（而非笼统说"使用了设计模式"）？
- 是否有关键接口和扩展机制的分析？

任一文件质量不达标 → 将其中的不足作为补充指令，重新派发子 Agent 补充分析。最多重试一次。

### 第六步：生成最终输出

#### 生成规则

1. 严格遵循输出模板结构
2. 技术栈分层展示（核心 → 辅助 → 基础设施），总计 8-15 项
3. 业务场景聚焦核心流程，避免罗列所有 CRUD 操作

5. 设计模式需标注具体应用位置（文件路径），不是笼统提及
6. 所有信息来自仓库实际内容，不编造；不确定的用 `<待补充>` 标记

## 关键规则

1. **并行优先**：第四步的三个子 Agent 必须同时派发，不允许串行
2. **中间文件落盘**：子 Agent 结果必须写入文件系统，不在上下文中口头传递
3. **只提取不编造**：所有数据必须来自仓库实际内容，不确定的用 `<待补充>` 标记
4. **三维度区分**：业务场景聚焦"做什么"，技术难点聚焦"怎么攻克"，设计决策聚焦"为什么这样做"
5. **一份产出**：每次调用处理一个仓库，生成一份知识文档
6. **语言跟随用户**：用户用中文提问就用中文输出
7. **不修改仓库**：只读分析，不对仓库做任何写入操作（中间文件写入 `.Poseidon/repo/` 而非仓库内）
8. **具体优于笼统**：每个结论必须有代码中的具体证据（文件路径、模式名、配置项），拒绝泛泛而谈
9. **粒度控制**：业务场景 2-5 个，技术难点 3-6 个，不过多也不过少

## 边界情况

- **非 Git 仓库**：没有提交历史 → 跳过 git log 分析，仅基于文件内容分析
- **空仓库 / 仅文档**：标注 `<待补充>` 并如实报告，不强行编造内容
- **大仓库 Clone 慢**：超过 2 分钟未完成 → 提示用户 Clone 较慢，询问是否改用本地路径
- **Monorepo**：用户指定了子目录则只分析该目录；否则分析整个仓库并标注"检测到 Monorepo 结构"
- **私有仓库**：Clone 需要认证 → 提示用户配置 SSH key 或 Personal Access Token
- **用户指定输出路径不存在**：自动创建目录
- **输出文件已存在**：询问用户覆盖还是追加时间戳重命名
- **小项目（< 20 文件）**：降低分析深度预期，在报告中说明项目规模限制

## 限制

- **不适用场景**：纯配置文件仓库、纯文档仓库、纯数据仓库（无代码逻辑可分析）
- **语言限制**：当前对主流语言（TS/JS、Python、Go、Java、Rust、C#）的项目分析效果最佳；小众语言或框架的项目分析深度受限
- **分析深度**：静态代码分析无法获取运行时行为、性能指标、线上数据等动态信息；业务场景推断依赖代码命名和注释质量
- **隐私注意**：分析的仓库内容会通过子 Agent 处理，敏感代码/私密仓库请确认环境安全性
- **输出定位**：本文档面向开发者理解项目全貌，不是架构评审报告也不是安全审计报告

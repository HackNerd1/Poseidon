# Worker Contract：技术架构与难点分析

## 输入

- 仓库路径：`.Poseidon/repo/<project_name>/`

## 输出

写入 `.Poseidon/repo/<project_name>/analysis/tech-challenges.md`

## 分析目标

### 1. 技术栈分层

- **核心依赖**：被多个模块引用的框架和库（如 React、Spring Boot、gRPC）
- **辅助依赖**：特定模块使用的工具库（如 excel 解析、图像处理）
- **基础设施**：数据库、消息队列、缓存、部署平台、CI/CD

从 `package.json`、`go.mod`、`Cargo.toml`、`requirements.txt`、`pom.xml` 等依赖文件提取。

### 2. 架构模式识别

识别以下模式（不限于此）：
- 分层架构（Controller → Service → Repository）
- 微服务 / 模块化单体
- 事件驱动 / CQRS / Event Sourcing
- 六边形架构 / 整洁架构 / DDD
- 插件系统 / 管道-过滤器

从目录结构、模块边界、依赖方向推断。

### 3. 技术难点（3-6 个）

重点寻找：
- **性能关键路径**：高并发处理、大数据量查询/写入、缓存策略、懒加载
- **数据一致性**：分布式事务、最终一致性方案、幂等设计、并发冲突处理
- **安全策略**：认证授权流程、数据加密、输入校验与注入防护、CSRF/CORS
- **可靠性**：容错机制、重试策略、熔断降级、超时控制、优雅关闭
- **复杂算法**：自定义协议实现、匹配/推荐/排序算法、状态机复杂度
- **跨领域关注点**：日志追踪、监控埋点、特性开关、灰度发布

### 4. 模块间通信

识别模块/服务间如何交互：RPC、REST、消息队列、事件总线、共享数据库。

## 质量标准

- 技术栈分三层，至少覆盖 5 项核心依赖
- 架构模式有具体代码证据（目录结构/关键文件）
- 每个技术难点包含："问题背景 + 解决方案 + 权衡取舍"三段，拒绝"用 Redis 缓存提升性能"这种一句话伪难点
- 难点必须有对应文件路径作为证据

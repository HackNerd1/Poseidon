# 面试准备清单生成工作流

## 概述

解析 Markdown 简历，提取专业技能、工作经历和项目经历，生成按技术栈和项目拆分的**多文件面试准备目录**。

**核心能力：**
- 每个技术栈独立成文，含简历锚定的准备要点、个性化模拟追问、联网收集的常见面试题
- 每个项目独立成文，含架构决策、难点攻克、量化溯源等深挖方向
- 可选 JD 输入：自动匹配 JD 要求与简历能力，生成定制化面试策略，在所有输出文件中注入 JD 相关提示

## 输出结构

```
interview/
├── index.md                  # 总索引（技术栈总览 + 项目链接 + 行为面试链接 + JD 匹配概览）
├── quick-reference.md        # 考前速查（电梯演讲 + 量化成果 + JD 关键词速查）
├── jd-match-analysis.md      # JD 匹配分析（仅在有 JD 时生成）
├── tech-stack/               # 按技术栈拆分
│   ├── react.md
│   ├── typescript.md
│   └── ...
├── projects/                 # 按项目拆分
│   ├── falcon.md
│   └── ...
└── behavioral/               # 行为面试故事库
    └── stories.md
```

输出到当前工作目录下的 `interview/` 目录（如用户指定了输出路径则使用指定路径）。

## 核心原则

1. **从简历出发**：所有知识点必须有简历中的技能或项目作为锚点，不凭空罗列
2. **项目驱动**：每个技术点关联用户真实项目经历，面试回答才能言之有物
3. **分层优先级**：核心技能（深度实践）> 辅助技能（项目经验）> 技术视野
4. **避虚就实**：不列纯概念题，而是结合项目追问
5. **分批生成**：技能文件逐个或分批生成，每批完成后报告进度
6. **联网补全**：每个技术栈文件生成时联网搜索最新常见面试题
7. **JD 感知**：有 JD 时，所有输出文件注入 JD 相关性标注和定制追问

## 工作流

### 第一步：获取简历与 JD

**简历获取：**
1. 用户提供了文件路径 → 直接读取
2. 未提供 → 扫描当前目录下 `*.md` 文件，筛选标题含"工程师""简历""CV"的文件让用户选择
3. 当前目录无匹配 → 询问用户提供路径

**JD 获取（可选）：**
1. 用户提供了第二个参数（JD 文本或 URL）→ 进入 JD 分析流程
2. 未提供 → 询问用户是否需要针对特定 JD 定制：
   ```
   是否需要针对特定岗位描述（JD）定制面试准备？
   如果有目标岗位的 JD，粘贴文本即可，我会分析匹配度并生成定制策略。
   ```
3. 用户选择不提供 → 跳过 JD 分析，按通用模式生成

确认后向用户展示：
```
找到简历文件：<文件名>
姓名：<从 H1 提取>
目标职位：<从 H1 提取>
JD：<已提供 / 跳过>
确认开始生成面试准备目录？
```

### 第二步：解析简历

从 `## 专业技能` / `## 工作经历` / `## 项目经历` 区域提取：

- **技能分组**：按 `**分组名：**` 识别分组，按顿号/逗号拆分为技能项
- **工作经历**：公司、职位、SARL bullet
- **项目经历**：项目名、技术栈、描述、亮点

展示解析结果供用户确认：
```
📚 已识别技术栈（共 x 项，y 个分组）
**分组名 A**：技能1、技能2、...
确认无误？可增删调整。
```

### 第三步：JD 匹配分析（有 JD 时执行）

**派发子 Agent 执行 JD 匹配分析：**

```
Agent(description: "JD匹配分析", prompt: "读取 references/jd-match-contract.md 的完整契约，对简历解析结果执行 JD 匹配分析，结果写入 interview/jd-match-analysis.md。简历技能：[列出已解析的技能和项目]。JD 文本：[用户提供的 JD 内容]")
```

等待子 Agent 完成后，向用户展示匹配结果摘要：

```
📊 JD 匹配分析完成

**强匹配**（x 项）：<列表> → 策略：证明深度
**部分匹配**（x 项）：<列表> → 策略：证明成长
**弱/无匹配**（x 项）：<列表> → 策略：诚实应答 + 迁移能力

确认后按此策略生成面试准备文件。
```

### 第四步：确定优先级分层

| 技能层级 | 判断依据 | 准备深度 |
|---------|---------|---------|
| 核心技能 (⭐⭐⭐) | 简历中列为"深度实践"，或有多个项目/经历支撑；JD 中为"强匹配"自动升一级 | 深度准备：核心原理、源码级问题、设计决策、性能优化 |
| 辅助技能 (⭐⭐) | 简历中列为"项目经验"，或有 1 个项目支撑 | 标准准备：常用 API、典型场景、与核心技能的配合 |
| 技术视野 (⭐) | 简历中列为"技术视野"，或无项目支撑 | 了解即可：核心概念、适用场景、与主力技术的差异 |

**JD 调整规则**：JD 中明确要求但简历匹配度低的技能，自动升一级准备深度。

向用户展示分层结果并确认。

### 第五步：创建目录结构与 index.md

```bash
mkdir -p interview/tech-stack interview/projects interview/behavioral
```

读取 `references/index-template.md`，按模板生成 `interview/index.md`。

有 JD 时，额外包含 JD 匹配概览表。

### 第六步：分批生成技术栈文件

**分批策略**：先核心技能（每批 2-3 个），再辅助技能（每批 3-5 个），最后技术视野（一批全部）。

每批完成后报告进度：
```
✅ 已完成：React、TypeScript、Electron（3/17）
⬜ 待生成：Next.js、JavaScript、HTML5/CSS3、...
```

**单个技术栈文件生成流程：**
1. 读取 `references/tech-stack-template.md` 获取模板
2. 联网搜索 `"<技能名> 面试题"` 和 `"<技能名> interview questions"`（有 JD 时追加 `"<技能名> <岗位关键词> 面试题"`）
3. 按模板生成文件，写入 `interview/tech-stack/<skill-name>.md`
4. 核心技能 ≥20 题、辅助技能 10-15 题、技术视野 3-5 题

### 第七步：生成项目深挖文件

对每个项目，读取 `references/project-template.md`，按模板生成 `interview/projects/<project-name>.md`。

有 JD 时，每个项目文件额外包含"JD 关联追问"方向。

### 第八步：生成行为面试故事库

读取 `references/behavioral-template.md`，从 SARL bullet 提炼 STARL 故事卡，写入 `interview/behavioral/stories.md`。

有 JD 时，标注每个故事可覆盖 JD 中的哪些软技能要求。

### 第九步：生成考前速查文件

读取 `references/quick-reference-template.md`，整合所有已生成内容，写入 `interview/quick-reference.md`。

有 JD 时，包含"JD 关键词速查"表。

### 第十步：验证与报告

输出最终报告：
```
✅ 面试准备目录已生成

**输出目录**：interview/
**目录结构**：
  - index.md（总索引）
  - quick-reference.md（考前速查）
  - jd-match-analysis.md（JD 匹配分析，如有 JD）
  - tech-stack/：x 个技术栈文件
  - projects/：x 个项目深挖文件
  - behavioral/：x 个行为面试故事

**覆盖统计**：
- 核心技能文件：x 个（含 ≥20 题/每个）
- 辅助技能文件：x 个
- 技术视野文件：x 个
- 项目深挖文件：x 个
- 行为面试故事：x 个

**JD 匹配总结**（如有 JD）：
- 强匹配 x 项 / 部分匹配 x 项 / 弱匹配 x 项
- 重点准备方向：<列出优先级最高的 3 项>

**下一步建议**：
- 面试前 1 周：通读 tech-stack/ + projects/ 全部内容
- 面试前 1 天：只看 quick-reference.md
- 面试前 30 分钟：背核心量化成果 + 电梯演讲 + JD 关键词速查
```

## 关键规则

1. **多文件输出**：不再生成单个巨型 MD，而是拆分为结构化多文件目录
2. **锚定简历**：所有知识点和追问锚定简历中的真实技能和项目，不凭空生成
3. **联网补全**：每个技术栈文件必须联网搜索最新面试题并整合
4. **分批生成**：技能文件分批生成，每批完成后报进度
5. **分层深度**：核心技能深度准备（≥20 题）、辅助技能标准准备（10-15 题）、技术视野了解即可（3-5 题）
6. **追问个性化**：模拟追问必须引用用户真实项目背景，不列通用题
7. **去重合并**：同一知识点在多个技能中出现时去重
8. **量化溯源**：对所有量化数据准备"这个数字是怎么算出来的"的回答
9. **确认再生成**：解析结果经用户确认后再进入生成步骤
10. **语言跟随用户**：用户用中文提问就用中文输出
11. **保留源文件**：不修改原始简历文件
12. **JD 全流程注入**：有 JD 时，从 index.md 到 quick-reference.md 的所有文件均包含 JD 相关性标注

## 边界情况

- **简历内容极少**：技能文件正常生成，项目深挖文件标记 `<待补充>`，提示用户先完善简历
- **技能项过多（>20）**：合并相近技能，按分组层级处理
- **无工作经历（应届生）**：跳过行为面试准备，强化项目深挖和技能知识点
- **跨行业求职**：技能映射关注可迁移能力，弱化行业特定术语
- **联网搜索无结果**：常见面试题章节标记「待联网补充」
- **用户指定输出路径**：在指定路径下创建 interview/ 目录
- **JD 与简历完全匹配**：正常生成，在报告中标注"高匹配度，重点准备深度追问"
- **JD 与简历几乎不匹配**：诚实标注差距，生成"差距说明"和"应答策略"文件，帮助用户准备转行/跨领域的解释话术
- **JD 为 URL**：使用 WebFetch 获取 JD 内容，失败时提示用户手动粘贴文本

## 限制

- **简历依赖**：输出质量完全取决于简历信息的完整度和准确性；简历空泛则面试准备也会空泛
- **联网搜索时效**：面试题来自搜索结果，可能存在过时或遗漏；建议用户结合目标公司的面经补充
- **JD 匹配非精确**：JD 分析基于关键词和语义匹配，不替代人工判断；JD 中隐含的"文化契合度"等软性要求无法自动评估
- **单次单简历**：每次调用处理一份简历，不支持多份简历并行
- **不适用场景**：非技术岗面试（如纯管理岗、销售岗）的技能准备效果有限；行为面试部分仍适用

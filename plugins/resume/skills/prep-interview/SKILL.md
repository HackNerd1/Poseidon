# 面试准备清单生成工作流

## 概述

解析 Markdown 简历，提取专业技能、工作经历和项目经历，生成按技术栈和项目拆分的**多文件面试准备目录**。每个技术栈独立成文，包含简历锚定的准备要点、个性化模拟追问、以及从网络收集的常见面试题；每个项目独立成文，包含架构决策、难点攻克、量化溯源等深挖方向。

## 输出结构

```
interview/
├── index.md                  # 总索引（技术栈总览表 + 项目链接 + 行为面试链接）
├── quick-reference.md        # 考前速查（电梯演讲 + 量化成果 + 关键词速查表）
├── tech-stack/               # 按技术栈拆分
│   ├── react.md
│   ├── typescript.md
│   ├── electron.md
│   └── ...
├── projects/                 # 按项目拆分
│   ├── falcon.md
│   ├── hermes.md
│   └── ...
└── behavioral/               # 行为面试故事库
    └── stories.md
```

输出到当前工作目录下的 `interview/` 目录（如用户指定了输出路径则使用指定路径）。

## 核心原则

1. **从简历出发**：所有知识点必须有简历中的技能或项目作为锚点，不凭空罗列面试题
2. **项目驱动**：每个技术点都关联用户的真实项目经历，面试回答才能言之有物
3. **分层优先级**：核心技能（深度实践）> 辅助技能（项目经验）> 技术视野
4. **避虚就实**：不列"请描述 React 生命周期"这种纯概念题，而是结合项目问"你在 XX 项目中为什么用 Hooks 而不是 Class？"
5. **分批生成**：技能文件逐个或分批生成，每批完成后报告进度，避免单次输出过长
6. **联网补全**：每个技术栈文件生成时，联网搜索该技术栈的最新常见面试题并整合到「常见面试题」章节

## 工作流

### 第一步：获取简历文件

1. **用户在调用时提供了文件路径** → 直接读取
2. **未提供文件路径** → 扫描当前目录下的 `*.md` 文件，筛选看起来像简历的文件（标题含"工程师""简历""CV"等），让用户选择；如只有一个匹配则直接确认
3. **当前目录无匹配** → 询问用户提供路径

确认文件后向用户展示：

```
找到简历文件：<文件名>
姓名：<从 H1 提取>
目标职位：<从 H1 提取>
确认从此文件生成面试准备目录？
```

### 第二步：解析简历

读取 Markdown 文件，提取以下信息：

#### 2.1 专业技能

从 `## 专业技能` 区域解析所有分组和技能项：

```
解析规则：
- 以 **分组名：** 或 **分组名** 开头的行识别为一个分组
- 分组名后的内容按顿号/逗号/斜杠拆分为技能项
- 每个技能项去空白、去多余符号
```

输出解析结果供用户确认：

```
📚 已识别技术栈（共 x 项，y 个分组）

**分组名 A**：技能1、技能2、...
**分组名 B**：技能3、技能4、...

确认无误？可增删调整。
```

#### 2.2 工作经历

从 `## 工作经历` 区域提取每段经历的公司、职位、业务背景、SARL bullet。

#### 2.3 项目经历

从 `## 项目经历` 区域提取每个项目的名称、技术栈、描述、亮点。

### 第三步：确定优先级分层

对每个技能项，确定层级和准备深度：

| 技能层级 | 判断依据 | 准备深度 |
|---------|---------|---------|
| 核心技能 (⭐⭐⭐) | 简历中列为"深度实践"，或有多个项目/经历支撑 | 深度准备：核心原理、源码级问题、项目中的设计决策、性能优化 |
| 辅助技能 (⭐⭐⭐) | 简历中列为"项目经验"，或有 1 个项目支撑 | 标准准备：常用 API、典型场景、与核心技能的配合方式 |
| 技术视野 (⭐) | 简历中列为"技术视野"，或无项目支撑 | 了解即可：核心概念、适用场景、与主力技术的差异 |

向用户展示分层结果并确认后，开始分批生成文件。

### 第四步：创建目录结构与 index.md

创建 `interview/` 目录及其子目录：

```bash
mkdir -p interview/tech-stack interview/projects interview/behavioral
```

生成 `index.md`，内容包含：

```markdown
# <姓名> — 面试准备索引

> 基于简历：<简历文件名>
> 生成时间：<YYYY-MM-DD>

## 目录结构
（列出完整目录树）

## 技术栈总览
| 优先级 | 技能 | 层级 | 关联项目数 |
（按优先级降序排列，每行链接到对应文件）

## 项目深挖
| 项目 | 类型 | 一句话定位 |
（每行链接到对应文件）

## 行为面试
| # | 故事 | 适用场景 |
（每行链接到对应文件）

## 面试节奏参考
- 面试前 1 周 / 1 天 / 30 分钟的建议
```

### 第五步：分批生成技术栈文件

**分批策略**：先核心技能（每批 2-3 个），再辅助技能（每批 3-5 个），最后技术视野（一批全部）。

每生成完一批，向用户报告进度：
```
✅ 已完成：React、TypeScript、Electron（3/17）
⬜ 待生成：Next.js、JavaScript、HTML5/CSS3、...
```

#### 单个技术栈文件模板

```markdown
# <技能名> 面试准备

> 优先级：<⭐⭐⭐/⭐⭐/⭐> <核心/辅助/视野> | 关联项目：<数量>（<项目名列表>）

---

## 准备要点

（简历锚定的准备要点，核心技能 5-8 个，辅助技能 3-5 个，技术视野 1-2 个）

- **<要点标题>** — <详细说明，关联简历中的具体项目场景>
- ...

## 模拟追问

（结合简历项目的个性化追问，核心技能 3-5 个，辅助技能 1-2 个）

1. "<结合简历项目的具体追问>"
2. ...

---

## 常见面试题

（联网搜索该技术栈的最新常见面试题，按子话题分组。核心技能 ≥20 题、辅助技能 10-15 题、技术视野 3-5 题）

### <子话题 1>

1. **<题目>**
2. **<题目>**
...

### <子话题 2>

1. **<题目>**
...
```

**联网搜索规则**：
- 对每个技术栈文件，使用 WebSearch 搜索 `"<技能名> 面试题"` 和 `"<技能名> interview questions"`
- 筛选高质量、高频出现的题目，去除重复和过时内容
- 按子话题组织（如 React 分为：基础与核心概念、Hooks 深度、状态管理、性能优化、并发特性等）
- 题目数量与层级匹配：核心技能 ≥20 题，辅助技能 10-15 题，技术视野 3-5 题
- 优先收录近年新出现的考点（如 React 19 新特性、TypeScript 5.x 新语法等）

### 第六步：生成项目深挖文件

对每个项目经历生成独立文件：

```markdown
# <项目名> — 项目深挖准备

> 一句话定位：<简历中的一句话描述>
> 类型：<个人项目 / 工作项目>

---

## 技术亮点

- <亮点 1>
- <亮点 2>
- ...

---

## 面试官关心的方向

### 1. 架构决策
**<核心问题>**
- <展开说明>

### 2. 难点攻克
**<核心问题>**
- <展开说明>

### 3. 量化验证
**<核心问题>**
- <展开说明>

### 4. 如果重来
**<核心问题>**
- <展开说明>

---

## 模拟深挖问题

1. "<具体追问>"
2. "<具体追问>"
3. ...
```

### 第七步：生成行为面试故事库

从工作经历的 SARL bullet 提炼 STARL 故事卡，写入 `behavioral/stories.md`：

```markdown
# 行为面试故事库

## 故事 N：<标题>

**关联 bullet**：<对应的简历 bullet>

**S**：<背景> | **T**：<任务> | **A**：<方案> | **R**：<成果> | **L**：<沉淀>

**适用问题**：
- "<常见行为面试问题>"
- ...
```

### 第八步：生成考前速查文件

生成 `quick-reference.md`：

```markdown
# 考前速查

## 核心技能关键词速查
| 技能 | 一句话定位 | 最可能被追问的点 |
（按优先级排列）

## 每个项目的 1 句话电梯演讲
（每个项目一行）

## 3 个最强的量化成果
（最有力的 3 条量化数据，附一句话说明）
```

### 第九步：验证与报告

输出最终报告：

```
✅ 面试准备目录已生成

**输出目录**：interview/
**目录结构**：
  - index.md（总索引）
  - quick-reference.md（考前速查）
  - tech-stack/：x 个技术栈文件
  - projects/：x 个项目深挖文件
  - behavioral/：x 个行为面试故事

**覆盖统计**：
- 核心技能文件：x 个（含 ≥20 题常见面试题/每个）
- 辅助技能文件：x 个
- 技术视野文件：x 个
- 项目深挖文件：x 个
- 行为面试故事：x 个 STARL 故事

**下一步建议**：
- 面试前 1 周：通读 tech-stack/ + projects/ 全部内容
- 面试前 1 天：只看 quick-reference.md
- 面试前 30 分钟：背核心量化成果 + 电梯演讲
```

## 关键规则

1. **多文件输出**：不再生成单个巨型 MD 文件，而是拆分为结构化的多文件目录
2. **锚定简历**：所有知识点和追问必须锚定简历中的真实技能和项目，不凭空生成
3. **联网补全**：每个技术栈文件必须联网搜索最新常见面试题并整合
4. **分批生成**：技能文件分批生成，每批完成后报进度，避免单次输出过长
5. **分层深度**：核心技能深度准备（≥20 题）、辅助技能标准准备（10-15 题）、技术视野了解即可（3-5 题）
6. **追问个性化**：模拟追问必须引用用户的真实项目背景，不列通用面试题
7. **去重合并**：同一知识点在多个技能中出现时去重，在核心技能下列出
8. **量化溯源**：对所有量化数据准备"这个数字是怎么算出来的"的回答
9. **确认再生成**：解析结果经用户确认后再进入生成步骤
10. **语言跟随用户**：用户用中文提问就用中文输出
11. **保留源文件**：不修改原始简历文件
12. **index.md 作为入口**：所有子文件通过 index.md 中的链接串联，方便快速检索

## 边界情况

- **简历内容极少**：可能只有技能列表无项目细节 → 技能文件正常生成，项目深挖文件标记 `<待补充>`，提示用户先完善简历
- **技能项过多（>20）**：合并相近技能，按分组层级处理，不在考前速查中列出所有项
- **无工作经历（应届生）**：跳过行为面试准备，强化项目深挖和技能知识点
- **跨行业求职**：技能映射时关注可迁移能力，弱化行业特定术语
- **用户提供了 JD**：将 JD 关键词与简历技能交叉对比，优先准备双方重叠的技能点
- **联网搜索无结果**：常见面试题章节标记「待联网补充」，不影响其他内容生成
- **用户指定输出路径**：在指定路径下创建 interview/ 目录，不做额外确认

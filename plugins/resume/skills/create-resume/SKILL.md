# 创建简历工作流

## 概述

根据用户提供的工作经历、项目经历、技能等信息，严格遵循《简历规范》（`references/简历规范.md`），生成一份 ATS 兼容、SARL 法则驱动的 Markdown 简历。

输出文件：`<姓名>-<目标职位>.md`，保存到当前工作目录。

## 岗位识别

在收集信息前，先确认用户的求职岗位类型。不同岗位的技能分组、动词选择、量化指标维度不同。

向用户确认：
```
你要创建什么岗位的简历？（如：后端工程师、前端工程师、数据分析师、产品经理、DevOps 工程师等）
```

根据岗位类型，在后续步骤中调整：
- **技能分组建议**：参考 `references/info-schema.md` 中各岗位的分组示例
- **量化维度侧重**：技术岗侧重性能/效率/质量指标；产品岗侧重用户增长/留存/转化指标；管理岗侧重团队规模/交付效率/业务增长
- **动词选择**：技术岗用"主导架构/设计/开发/重构/优化"；产品岗用"主导/规划/推动/定义/上线"

## 核心规范速查

以下关键规则摘自《简历规范》，生成简历时必须逐条遵守。完整规范见 `references/简历规范.md`。

### 结构规则

| 规则 | 说明 |
|------|------|
| 模块顺序 | 基本信息 → 个人简介 → 专业技能 → 教育背景 → 工作经历 → 项目经历 |
| 页数控制 | 应届 1 页，中级 1-2 页，高级 ≤2 页（Markdown 约 80-150 行） |
| 6 秒测试 | 前半页必须可见：角色定位 + 最强技能栈 + 一个量化亮点 |

### 个人简介公式

> **[角色] + [核心技能/年限] + [一个量化亮点] + [求职方向]**

禁止套话："性格开朗""工作认真""学习能力强""团队协作好"。

### 专业技能规则

- 按领域分 3-5 组，共 12-20 项
- 每组按匹配度排序，最强放最前
- 第一组放目标岗位最关心的技能
- 禁止技能进度条/百分比/图标
- 用"深度实践"代替"精通"，"项目经验"代替"熟练"

### 工作经历 — SARL 法则

> **[动词] + [技术/业务方案] + [量化成果] + [方法论沉淀]**

| 维度 | 含义 |
|------|------|
| **S** (Situation) | 项目背景与挑战 |
| **A** (Approach) | 方案与决策过程 |
| **R** (Result) | 量化成果 |
| **L** (Lessons) | 方法论沉淀与可复用经验 |

每段经历 ≥1 个量化成果 + 方法论沉淀。

### 常见减分项（必须避免）

- ❌ "参与/协助了 XX" → ✅ "主导/负责/独立负责"
- ❌ "熟练使用各种技术/工具" → ✅ 列出具体技能并嵌入 bullet
- ❌ "精通 XX" → ✅ "深度实践"，用项目细节证明
- ❌ "优化了 XX" → ✅ 加具体量化指标
- ❌ 技能区罗列 20+ 项无分组 → ✅ 精选 12-20 项，分 3-5 组
- ❌ 个人简介写套话 → ✅ 用具体数据和差异化表述

## 工作流

### 第一步：解析输入

判断用户输入：

1. **提供了文件路径**（`.json` / `.yaml` / `.yml`）→ 读取并解析文件，进入第二步
2. **提供了文件路径但文件不存在** → 提示用户文件不存在，转为对话收集模式
3. **未提供文件** → 直接进入对话收集模式
4. **用户直接粘贴了信息文本** → 从中提取可用字段，剩余部分对话补充

如果用户提供了文件，读取后对照 `references/info-schema.md` 中的 Schema 解析，标记已获取和缺失的字段。

### 第二步：对话收集缺失信息

先确认岗位类型（见"岗位识别"），再逐 section 检查信息完整性，只询问缺失或模糊的字段。按以下顺序收集：

#### 2.1 基本信息

询问缺失项：
- 姓名（必填）
- 手机（必填）
- 邮箱（必填）
- GitHub（选填但强烈建议）
- 博客/作品集（选填）
- 工作年限（必填，用于确定简历级别）
- 求职方向（必填，如"后端工程师""数据分析师""产品经理"）

#### 2.2 个人简介

如果用户未提供个人简介，基于已收集的信息草拟一版，让用户确认或修改。

草拟公式：`[年限] 年 [角色] + [核心技能] + [一个量化亮点（如有）] + [求职方向]`

示例输出给用户确认：
```
我根据你的信息草拟了个人简介：
"5 年后端工程师，深耕 Java 微服务与高并发系统设计，主导过日均千万级
请求的支付网关架构。具备 AWS 云原生与 Kubernetes 容器化实战经验。"
是否合适？你可以直接修改。
```

#### 2.3 专业技能

根据岗位类型给出对应的分组建议（参考 `references/info-schema.md`），引导用户提供：

```
请按以下分组提供你的专业技能（不确定熟练度的可以留空）：

<根据岗位动态生成分组建议>

每组 2-5 项，总共 12-20 项。按熟练度和与目标岗位的匹配度排序。
```

如果用户提供了 JD（第三步会用到），优先展示 JD 中高频的技能关键词作为建议。

#### 2.4 教育背景

收集：学校、专业、学位、起止时间。

#### 2.5 工作经历

对每段工作经历收集：

- 公司名称（必填）
- 职位（必填）
- 起止时间（必填，格式 YYYY.MM）
- 项目/业务背景（必填，1-2 句话，用于 SARL 的 S）
- 你的角色与职责（必填，用于 SARL 的 T）
- 主要成果/亮点（必填，至少 2 条。引导用户用"动词 + 方案 + 量化成果"的格式描述）

引导话术：
```
请描述你在 [公司] 担任 [职位] 期间的一段核心经历。
不用写得多漂亮，告诉我：
1. 当时在做什么业务/产品？（S - 背景）
2. 你负责什么？（T - 任务）
3. 你用了什么方案/方法？（A - 方案）
4. 取得了什么成果？有没有具体数据？（R - 结果）
5. 有什么经验可以沉淀复用？（L - 方法论）

我来帮你润色成 SARL 格式。
```

**重要**：收集完成后，在第五步生成时会自动将原始描述改写为 SARL 格式。

#### 2.6 项目经历

对每个项目收集：

- 项目名称（必填）
- 在线链接（GitHub / 在线预览 / 文档，选填但强烈建议）
- 技术栈/工具（必填，3-5 项）
- 项目一句话描述（必填）
- 亮点（至少 2-3 条，含量化数据）

引导话术：
```
请描述你的项目 [项目名称]：
1. 这个项目是做什么的？（一句话）
2. 用了哪些核心技术/工具？
3. 你做了什么关键工作？（2-3 个亮点）
4. 有什么量化成果？
5. 有在线链接吗？（GitHub / 预览地址 / 文档）
```

### 第三步：JD 匹配分析（可选）

询问用户是否需要针对特定 JD 定制简历：

```
是否需要针对特定岗位描述（JD）优化简历？
如果是，请粘贴目标 JD 文本。我会提取关键词并调整简历重点。
```

如果用户提供了 JD：

1. **提取 8-12 个高频关键词**：从 JD 中识别重复出现的术语，按出现频率和重要性排序
2. **对齐目标职位名**：确保简历中的角色名与 JD 一致
3. **生成关键词覆盖清单**：列出 JD 核心词 → 简历对应位置，确保每个词都被自然覆盖
4. **调整技能排序**：第 1 组技能优先展示 JD 最看重的领域
5. **标注重点 bullet**：标记与 JD 匹配度最高的 4-6 条 bullet，在生成时优先使用 JD 关键词

将 JD 分析结果展示给用户确认后再进入生成步骤。

### 第四步：确认信息完整性

汇总所有已收集的信息，以结构化形式展示给用户做最终确认：

```
📋 信息汇总确认

**基本信息**
- 姓名：xxx | 手机：xxx | 邮箱：xxx
- 工作年限：x 年 | 求职方向：xxx
- GitHub：xxx

**专业技能**（x 组，共 x 项）
- 分组1：xxx
- 分组2：xxx
- ...

**教育背景**
xxx 大学 | xxx 专业 | 本科 | 2018 - 2022

**工作经历**（x 段）
1. xxx 公司 | xxx 职位 | 2022.03 - 2024.06
   - 亮点数：x 条

**项目经历**（x 个）
1. xxx 项目 | 技术栈：xxx
   - 亮点数：x 条

是否确认开始生成简历？你可以：
- 回复"确认"直接生成
- 指出需要修改的部分
- 补充遗漏的信息
```

### 第五步：生成简历

严格遵循 `references/resume-template.md` 模板格式生成简历。

#### 5.1 生成规则

1. **基本信息**：简洁一行展示（姓名、手机、邮箱用分隔符隔开），GitHub/博客另起一行
2. **个人简介**：严格按公式，2-3 句话，禁止套话
3. **专业技能**：按领域分 3-5 组，每组一行。有 JD 时第一组对齐 JD。禁止进度条/百分比。用"深度实践""项目经验""技术视野"代替"精通""熟练""了解"
4. **教育背景**：简洁一行
5. **工作经历**：使用 SARL 改写。每条 bullet 结构：
   - 动词开头（主导/设计/开发/重构/优化）
   - 嵌入 1-2 个专业关键词
   - 1 个量化成果（原始数据没有则用 `<待补充>` 标记）
   - 方法论沉淀（L 维度）
   - 每条 bullet 控制在 1-2 句，聚焦一个主题
6. **项目经历**：每个项目含名称、链接、技术栈、一句话简介、2-3 个亮点

#### 5.2 ATS 优化（有 JD 时生效）

- 自然嵌入 JD 关键词到 bullet 中（不堆砌）
- 每条 bullet 聚焦 1-2 个核心技能，体现深度
- 标题使用标准名称（专业技能 / 工作经历 / 项目经历 / 教育背景）
- 不写"Proficient in various technologies"类空话

#### 5.3 量化数据标记

- 用户提供了数据 → 直接使用
- 用户提到但无精确数据 → 引导用户补充，先用 `<待补充: 具体数值>` 标记
- 完全没有量化数据 → 标记 `<待补充: 量化成果>`，并在最终报告中提示

#### 5.4 阶段适配

根据工作年限调整侧重点：

| 年限 | 侧重点 |
|------|--------|
| 应届/初级 (0-1) | 学习能力、实习/校园项目、专业热情、GitHub 活跃度 |
| 中级 (1-3) | 项目深度、独立解决问题能力、技能成长轨迹 |
| 高级 (3-5) | 架构设计、跨团队协作、业务理解、方法论沉淀 |
| 资深 (5+) | 行业影响力、团队领导力、从 0 到 1 经验、业务增长贡献 |

### 第六步：质量检查

生成后逐项自检，发现问题自动修正：

- [ ] 前半页可见：角色定位 + 最强技能栈 + 一个量化亮点（6 秒测试）
- [ ] 个人简介无套话（"性格开朗""学习能力强"等）
- [ ] 专业技能按领域分 3-5 组，共 12-20 项，第一组对齐 JD（如有）
- [ ] 无"精通""参与""各种"等减分措辞
- [ ] 每段工作经历 ≥1 个量化成果 + 方法论沉淀（SARL 的 L 维度）
- [ ] 项目描述含在线链接（如有）
- [ ] 无技能进度条/百分比/照片/图标/多栏布局
- [ ] 文件命名：`姓名-目标职位.md`

### 第七步：输出与报告

保存文件到当前工作目录：`<姓名>-<目标职位>.md`

向用户报告：

```
✅ 简历已生成

**输出文件**：<姓名>-<目标职位>.md（当前目录）

**包含内容**：
- 基本信息
- 个人简介（x 年经验）
- 专业技能（x 组，共 x 项）
- 教育背景
- 工作经历（x 段，共 x 条 SARL bullet）
- 项目经历（x 个，共 x 条亮点）

**待补充清单**：
- <列出所有带 `<待补充>` 标记的项>

**下一步建议**：
- 补充待完善的数据
- 如需针对特定 JD 优化，可以重新运行并提供 JD 文本
```

## 关键规则

1. **规范先行**：所有生成内容必须符合《简历规范》（`references/简历规范.md`），不确定的地方回查对应文件
2. **岗位感知**：根据用户岗位动态调整技能分组建议、量化维度侧重和动词选择
3. **技能诚实**：不引入用户无项目支撑的工具/框架，凡无项目经验的技能应主动剔除（详见简历规范第八节）
4. **项目筛选**：优先展示最新/最有技术价值的项目，较早期项目在无特定 JD 要求时不纳入（详见简历规范第八节）
5. **只问缺失项**：不重复询问用户已在文件中提供的信息
6. **确认再生成**：信息汇总后必须经用户确认才能进入生成步骤
7. **不编造数据**：所有量化数据必须来自用户输入，无法获取的用 `<待补充>` 标记
8. **SARL 完整**：每段工作经历必须覆盖 S/A/R/L 四维度，至少含 1 个量化成果
9. **语言跟随用户**：用户用中文提问就用中文输出
10. **一次一份**：每次调用生成一份简历

## 边界情况

- **信息严重不足**：用户只提供了姓名和少量信息 → 分步引导，每次只问 1-2 个 section，不一次抛出所有问题
- **工作经历过多（>5 段）**：只保留最近 3-4 段（中级）或 4-5 段（高级/资深），非核心经历合并或删减
- **项目经历过多（>5 个）**：挑选最核心的 2-3 个，其余列出名称+技术栈+链接即可
- **无工作经历（应届生）**：将项目经历/实习经历提升到工作经历位置，侧重点调整为学习能力和专业热情
- **跨行业转行**：引导用户挖掘可迁移技能，弱化行业特定经验，突出核心能力
- **用户中途取消**：已收集的信息不保留，不生成不完整的简历
- **JSON/YAML 格式错误**：提示具体解析错误位置，建议用户修复后重试或转为对话收集

# 前端工程师简历规范

> 综合 2025-2026 年国内外前端简历最佳实践、ATS 筛选标准、SARL 法则及优秀案例。

---

## 一、简历整体结构

### 1.1 推荐模块顺序

```
1. 基本信息    — 姓名、手机、邮箱、GitHub、博客/作品集
2. 个人简介    — 2-3句话，技术栈+核心亮点+求职方向
3. 技术栈      — 分层分类：精通 → 熟练 → 了解
4. 教育背景    — 学校/专业/时间
5. 工作经历    — 时间倒序，SARL 法则 + 量化成果
6. 项目经历    — 2-3个核心项目，含在线链接
```

### 1.2 格式规范

| 规则 | 说明 |
|------|------|
| **页数** | 应届 1 页，中级 1-2 页，高级 ≤2 页 |
| **格式** | PDF，文件名 `姓名-前端工程师.pdf` |
| **字体** | 中文用系统默认，英文用 Arial/Calibri，正文 10-12pt，姓名 14-16pt |
| **布局** | 单栏布局（ATS 兼容），不用图标/技能进度条/照片 |
| **颜色** | ≤2 种颜色，不用花哨模板 |

### 1.3 6 秒测试

招聘者平均只花 **6 秒** 扫一份简历。前 1/2 页必须让 recruiter 立即看到：

1. **你是什么类型的开发者**（前端 / 全栈偏前端）
2. **最强技术栈**（从 Summary + 技能区顶部获取）
3. **一个量化亮点**（最近一段经历的第一个 bullet）

三个都不在前半页 → 简历必定 ATS 和人工双失败。

## 二、个人简介写法

### 公式

> **[角色] + [核心技能/年限] + [一个量化亮点] + [求职方向]**

### 示例

```
4 年前端/全栈工程师，深耕 Electron 桌面客户端与 React 生态，熟悉 CI/CD 与安全加固。
具备 Python 后端与 AWS 云服务实战经验。另有 Android 与 HarmonyOS App 开发及上线经验。
```

- **要点**：突出差异化（如全栈能力、跨端经验），避免套话（"性格开朗""工作认真"）

## 三、技术栈展示

### 3.1 按领域分组法（推荐）

> **2026 年主流做法**：按职能分组，而非按熟练度分层。"精通/熟练/了解"主观且 ATS 无法解析。

```
语言：TypeScript、JavaScript (ES6+)、HTML5、CSS3、Python
框架：React、Next.js、Electron、Vue3
状态管理：Redux、Zustand、React Query
测试：Jest、React Testing Library、Cypress、Playwright
工具链：Webpack、Vite、Git、GitHub Actions、Docker
```

- 每组按与目标 JD 的匹配度排序，最强放最前
- 数量建议：**12-20 项**，分 3-5 组
- 第一组放目标岗位最关心的技术（通常是框架/语言）

### 3.2 分层分类法（备选）

如果熟练度差异确实很大，可用文字描述（不用进度条）：

```
深度实践：React、TypeScript、Electron、JavaScript (ES6+)、HTML5/CSS3
项目经验：Next.js、Node.js、Webpack、Vite、Git、CI/CD
技术视野：Flutter、WebGL、Python/FastAPI、AWS
```

> 注意：用"深度实践"代替"精通"，用"项目经验"代替"熟练"，避免主观评级词。

### 3.3 2025-2026 热门关键词

- **框架**: React (Hooks/Server Components)、Next.js、Vue3 (Composition API)
- **构建**: Vite、Turbopack、Webpack 5
- **类型**: TypeScript（泛型、条件类型）
- **测试**: Vitest、Playwright
- **无障碍**: WCAG 2.1 AA、ARIA、a11y、屏幕阅读器测试
- **设计系统**: 组件库、Design Tokens、Storybook、Figma-to-code
- **趋势**: AI 辅助开发、SSR/边缘计算、微前端、Core Web Vitals（LCP/CLS/INP）

> **2026 年特别关注**：无障碍（a11y）已成硬性要求，设计系统经验是 #1 差异化因素。AI 辅助开发要展示流程改进（"用 AI 加速了什么"），而非仅罗列工具名。

## 四、工作经历描述 — SARL 法则

> 2026 年主流已从 STAR 升级为 **SARL**，增加了"经验沉淀"维度，体现技术决策背后的思考深度。

### 4.1 核心公式

> **[动词] + [技术方案] + [量化成果] + [方法论沉淀]**

### 4.2 SARL 五要素

| 维度 | 含义 | 示例 |
|------|------|------|
| **S** (Situation) | 项目背景与挑战 | "旧平台首屏 5.8s，用户流失严重" |
| **T** (Task) | 你的角色与目标 | "作为前端负责人，目标首屏 ≤2s" |
| **A** (Approach) | 技术方案与决策过程 | "引入 SSR + 代码分割 + CDN + WebP" |
| **R** (Result) | 量化成果 | "LCP 从 3.8s 降至 2.2s，包体积减少 54%" |
| **L** (Lessons) | 方法论沉淀与可复用经验 | "沉淀了前端性能优化清单，后续 3 个项目复用，平均 Lighthouse 90+" |

> **L 是关键差异化点**：不只看你做了什么，还看你能否抽象出可复用的方法论。面试官通常会追问"你从中学到了什么"。

### 4.3 优化对比

| 差 | 优 |
|----|----|
| "参与电商网站开发" | "主导 React 18 + TypeScript 电商平台重构，首屏从 3.2s 降至 1.8s (LCP 65→92)，转化率提升 15%。沉淀了 SSR 性能优化 checklist，后续 2 个项目复用" |
| "负责后台管理系统开发" | "独立开发订单管理模块，虚拟滚动处理 10 万+数据流畅展示，客诉率下降 40%。抽象虚拟滚动 Hooks 库被团队 3 个项目采用" |
| "提升了系统性能" | "通过 WebWorker + 缓存策略，QPS 从 1k 提升至 1.5k，平均响应时间下降 96%。输出了《前端性能优化手册》，成为团队 code review 标准" |
| "开发了 XX 功能" | "搭建微前端架构，子系统复用率达 75%，开发效率提升 40%。制定了微前端接入规范，跨团队落地用时从 2 周缩短至 3 天" |
| "使用 Copilot 辅助开发" | "搭建 AI 辅助代码审查流程（Copilot + 自定义 ESLint 规则），PR review 时间从 2 天缩短至 4 小时，线上 bug 率下降 30%" |

### 4.4 动词参考

| 级别 | 推荐动词 |
|------|---------|
| 主导/架构 | 主导、设计、架构、搭建、从零构建 |
| 核心开发 | 开发、重构、优化、实现、集成 |
| 参与/协作 | 参与、协作、维护、支持 |

## 五、量化指标参考

### 5.1 可量化维度

| 类别 | 指标示例 |
|------|---------|
| **Core Web Vitals** | LCP（最大内容绘制）、CLS（累计布局偏移）、INP（交互到下次绘制） |
| **性能** | 首屏时间、包体积、FPS、Lighthouse/PageSpeed 评分 |
| **效率** | 构建时间、部署频率、开发周期、组件复用率、PR review 时间 |
| **质量** | Bug 率、错误率、测试覆盖率、E2E 通过率、线上 type error 数 |
| **业务** | 转化率、跳出率、用户留存、客诉率、成本降低、日均订单量 |
| **无障碍** | WCAG 合规等级、axe-core 检测通过率、屏幕阅读器兼容覆盖率 |

### 5.2 Core Web Vitals 专项（前端必备）

前端是少数能直接量化用户体验的岗位。2026 年简历标配：

```
"LCP 从 4.2s 降至 2.2s（Next.js SSR + 图片优化 + 关键 CSS 内联），Lighthouse 42→94"
"CLS 从 0.25 降至 0.02（预设图片尺寸 + 骨架屏 + 字体预加载）"
"INP 从 350ms 降至 120ms（代码分割 + Web Worker 卸载主线程任务）"
```

### 5.3 量化写法模板

- "通过 __ 技术方案，将 __ 从 __ 提升/降低至 __ "
- "减少 __% 安装包体积"
- "节省 __% 成本"
- "开发效率提升 __%"

## 七、ATS 适配要点

**75% 的简历从未到达人工 recruiter**，在 ATS 阶段即被淘汰。2026 年主流 ATS（Greenhouse、Lever、Workday）已升级为**语义匹配**，而非简单关键词计数。

### 7.1 ATS 核心原则

- 从 JD 提取 8-12 个高频关键词，确保简历自然覆盖
- **上下文和相关性 > 关键词密度** — 堆砌关键词会触发 rejection
- 用标准术语（如 "React" 而非 "React 框架"）
- 单栏布局，不用表格/文本框/图片/图标
- 不要用技能进度条/图表（机器无法识别）
- 标题用标准名称："Experience""Education""Skills"（不用创意变体）
- **PDF 或 .docx**：部分 ATS 仍偏好 .docx（文本提取更准）；PDF 保留排版更好

### 7.2 前端岗位 ATS 高频关键词清单

按在 JD 中出现频率分级：

**80%+ 的 JD 包含**：
React、JavaScript、TypeScript、HTML/CSS、响应式设计、Git

**50-80% 的 JD 包含**：
Next.js、Redux、REST API、GraphQL、Jest、无障碍 (WCAG/ARIA)、CI/CD、Tailwind CSS、Figma

**30-50% 的 JD 包含**：
Webpack、Vite、Storybook、Cypress、Playwright、性能优化、设计系统、SSR/SSG

**新兴差异化词（含金量高）**：
微前端、WebSocket、Service Worker、i18n、Monorepo (Turborepo/Nx)、Core Web Vitals、AI 辅助开发、边缘计算

### 7.3 关键词的正确嵌入 vs 错误堆砌

| 错误（堆砌） | 正确（嵌入） |
|------|------|
| 技能区写 "React, Next.js, TypeScript, Webpack, Vite, Jest, Cypress, Docker, AWS..." 连写 20 个 | 在 bullet 中自然嵌入："用 Next.js SSR + TypeScript 泛型 API 层重构，Lighthouse 65→94" |
| 每个 bullet 塞 5+ 技术名词 | 每条 bullet 聚焦 1-2 个核心技术，体现深度 |
| "Proficient in various web technologies" | 不写这种话，直接列出具体技术并用项目经历证明 |

> **关键原则**：如果简历上列了某个技术，必须能在面试中被深度追问。列 5 个框架却没深度项目支撑 → 面试官默认你一个都不精通。

## 八、补充约束

> 补充于 2026-05-30，用于约束简历生成输出。

### 8.1 技能列表诚实原则

不引入不熟悉的工具/框架。技能列表中的每一项都必须有明确的项目经历或工作经历支撑，确保面试时可被深度追问。凡无项目支撑的技能，应主动剔除，不因「热门」而强行列入。

### 8.2 项目筛选优先级

当项目经历过多需要筛选时，按以下优先级排序：

1. **最能体现技术价值** — 技术深度、架构决策、量化成果突出的项目
2. **最符合主流趋势** — 技术栈与当前行业主流对齐的项目
3. **最符合 JD** — 与目标岗位描述匹配度最高的项目
4. **最新** — 优先展示最近期项目

较早期项目（如距今超过 4 年），在无特定 JD 要求时不再纳入简历。

### 8.3 教育背景位置

教育背景放在「专业技能」之后、「工作经历」之前（中文简历常见布局）。此顺序与英文简历（教育背景放最后）不同，中文简历中教育背景前置更有利于 recruiter 快速了解学历层次。

## 九、常见减分项

### 9.1 措辞减分项

| 错误 | 问题 | 修正 |
|------|------|------|
| "参与 / 协助了 XX 项目" | 无 ownership 信号，ATS 无法评分 | 用"主导 / 负责 / 独立开发"替换 |
| "熟练使用各种前端技术" | "各种" = ATS 空值 | 列出具体技术并在 bullet 中嵌入 |
| "热衷构建优美的 Web 应用" | 空洞套话，无法量化 | 删除，改用具体成果和数据 |
| "学习能力强 / 团队协作好" | ATS 无法匹配到 JD 要求 | 通过项目行为体现，不直接宣称 |
| "精通 HTML/CSS/JavaScript" | "精通"滥用 → 可信度归零 | 写"深度实践"，用项目细节证明 |
| "用 React 开发了前端页面" | 没有技术深度，任何候选人都能写 | 写明技术决策 + 量化产出 |
| "优化了页面性能" | 没有量化，等于没写 | 加具体指标："LCP 3.8s → 2.2s" |
| 技能区罗列 20+ 技术 | 面试官默认你一个都不精通 | 精选 12-20 项，分 3-5 组，用项目证明 |

### 9.2 格式减分项

| 错误 | 修正 |
|------|------|
| 技能罗列无主次 | 按与目标 JD 的匹配度排序，最强放最前 |
| 个人简介写套话 | 用具体数据和差异化表述 |
| 技能用进度条/百分比 | 用文字描述（按领域分组），不用任何图形 |
| "精通"一词滥用 | 用"主导重构""深度实践"替代 |
| 项目描述无业务背景 | 关联业务价值（"支撑日均 10 万+订单"） |
| 堆砌无关技能 | 根据 JD 剔除不相关技术栈 |
| 同一份简历海投 | 针对每个岗位微调关键词（5 份精准投递 > 50 份海投） |
| 罗列"沟通能力""团队协作" | 通过项目/工作中的行为体现 |
| HTML/CSS 放技能区最前 | 默认给人初级印象；优先展示 React / TypeScript / 框架 |
| 文件名无姓名和职位 | 固定格式：`姓名-前端工程师.pdf` |

## 十一、投递工作流

> 5 份精准投递（ATS 匹配度 90%+）>>> 50 份通用海投（匹配度 40%）。

### 每次投递前按以下流程操作：

1. **提取 JD 关键词**：从目标岗位 JD 中抓取 8-12 个重复出现的术语
2. **对齐职位名称**：确保简历中的目标角色名与 JD 一致（"Frontend Engineer"≠"Web Developer"）
3. **定制 Summary**：改写个人简介，围绕该岗位的核心技术栈和你的最强亮点
4. **重排技能顺序**：第 1 组技能必须是该 JD 最看重的领域
5. **重写前 4-6 条 bullet**：用 Technology + Decision + Outcome + Lessons 结构
6. **验证关键词覆盖**：对照 JD 逐项检查，确保每个核心词都出现在具体的项目 bullet 中
7. **ATS 跑分**：用 ATS checker 工具（如 ResumeAdapter、Jobscan）检测匹配度
8. **输出文件命名**：`姓名-前端工程师.pdf`

## 十二、2026 新兴趋势（前瞻）

### 11.1 AI 筛选时代的关键变化

- **AI-to-AI 优化**：同一 LLM 生成的简历被同一 LLM 筛选时，通过率高 23-60%。建议准备多版简历优化不同 ATS 模型
- **语义匹配 > 关键词计数**：Greenhouse、Lever 等已升级语义匹配，关键词的上下文比出现次数更重要
- **欧盟 AI 法案**将 AI 招聘列为"高风险"，要求可解释性和审计追踪—ATS 对真实、有据可查的叙述更友好

### 11.2 简历之外：多模态评估正在兴起

- **交互式 AI 作品集**：个人网站嵌入 AI chatbot，面试官可直接提问你的项目经历和决策逻辑
- **GitHub 提交历史被纳入筛选**：代码质量、commit 频率、开源贡献成为评估维度
- **LinkedIn 技能验证**：基于实际工具使用数据自动生成验证徽章，而非一次性考试
- **视频回答**：部分公司要求录制技术问题作答，作为初筛补充

> **核心启示**：简历仍然重要，但已不再是决定性因素。将简历视为"入口"而非"全部"，配合 GitHub 活跃度 + 在线作品集 + AI 作品集构建完整个人品牌。

## 十三、自检清单

投递前逐项确认：

- [ ] 页数 ≤ 2 页，PDF 或 .docx 格式
- [ ] 前 1/2 页可见：角色定位 + 最强技术栈 + 一个量化亮点（6 秒测试）
- [ ] 姓名/联系方式在第一屏可见
- [ ] 个人简介包含年限 + 核心技能 + 量化亮点（非套话）
- [ ] 技术栈按领域分 3-5 组，共 12-20 项，第一组对齐 JD
- [ ] 每段工作经历有至少 1 个量化成果 + 方法论沉淀（SARL）
- [ ] 项目描述含在线链接（GitHub / Storybook / 在线预览）

- [ ] 包含 Core Web Vitals 指标（LCP / CLS / INP）
- [ ] 无技能进度条、无照片、无花哨模板、无多栏布局

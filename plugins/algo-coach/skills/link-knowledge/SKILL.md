# 知识点关联工作流

## 概述

读取用户提供的算法题文件，识别其算法类别，在 hello-algo.com 上找到对应的知识点页面，并在文件中添加解题思路和学习笔记，把孤立的题目锚定到系统化的知识图谱上。

## 工作流

### 第一步：读取题目文件，提取信息

读取文件内容，从中提取：

1. **题目标题**：从文件名、注释、或文档字符串中获取
2. **题目描述**：从文件顶部的注释块中获取
3. **现有解法代码**：提取实际的代码实现
4. **编程语言**：从文件扩展名和语法判断

以下信息如果无法从文件中直接提取，**向用户询问**：

- **题目难度**（简单/中等/困难）—— 如果文件名中没有标注
- **算法类别**（贪心/DP/双指针/回溯/...）—— 如果用户没说明，且从代码中也无法确定
- **题目来源**（LeetCode/牛客/AcWing/...）—— 如果用户没有说明

### 第二步：确定算法类别

首选用户明确告知的类别。否则根据题目描述和代码中使用的算法技巧推断，读取 `references/category-mapping.md` 找到对应的 hello-algo 章节。

常见判断线索：
- 每步做局部最优选择，不回退 → 贪心
- 状态转移方程 + 最优子结构 → 动态规划
- 左右指针从两端向中间收缩 → 双指针
- 穷举 + 剪枝回退 → 回溯
- 有序区间 + O(log n) 查找 → 二分查找

如果仍然不确定，向用户确认类别。

### 第三步：搜索 hello-algo 知识点（精确 → 近似）

不是所有题目都能在 hello-algo 找到原题，找不到是正常情况。此时需要自行判断题目所属类别，找出最接近的知识点。

#### 3.1 精确搜索

用题目标题中的关键词搜索 hello-algo 中对应的知识页：

1. **优先搜索**：`site:hello-algo.com <题目关键词>`
2. **备选搜索**：`site:github.com/krahets/hello-algo <题目关键词>`

#### 3.2 近似匹配（精确搜索未命中时）

如果精确搜索没有命中独立专题页，按以下流程处理：

1. **确认主类别**：回顾第二步的判断，确定题目属于哪个算法类别
2. **找最接近的子页面**：在对应章节下搜索与该题思路最接近的子页面（如"接雨水"最接近第 15 章贪心 §15.3 最大容量问题、第 4 章数组与链表 § 双指针技巧）
3. **跨章节关联**：如果题目涉及多个类别的技巧（如双指针 + DP），可同时引用多个章节的知识点，在解题思路中说明各部分对应哪个知识点
4. **兜底**：若章节下也没有相关子页面，使用章节首页 URL，并在报告中说明"未找到精确匹配，使用最近章节页"

#### 3.3 判断最近知识点的标准

从以下维度判断哪个知识点最接近：
- **解题策略相似度**：核心思路是否类似（如"接雨水"和"最大容量问题"都用双指针从两端收缩）
- **数据结构重合度**：是否使用相同的数据结构
- **问题模式**：是否属于同类问题变种（如区间问题、子序列问题、路径问题）

### 第四步：获取知识点内容

获取 hello-algo 知识页内容（精确匹配页 or 最近似页面），重点关注：算法思想、解题步骤、正确性证明、复杂度分析、**该算法类别的适用条件**（如贪心选择性质、最优子结构等——这些会成为"适用依据"字段的出处）。

如果使用近似匹配，额外获取对应章节首页或其他相关子页面的内容作为补充。

如果网站请求失败（如 429 限流），使用搜索摘要作为备选信息来源。

### 第五步：生成解题思路并写入文件

在题目描述注释和代码之间插入结构化的解题思路注释。注释格式适配当前文件的编程语言（`//` 或 `/* */` 或 `#` 等）。

注释块包含以下内容：

- **算法类别**：类别名。如果涉及多个类别，列出主类别和辅助类别
- **hello-algo 知识点**：节标题 + URL。精确匹配到一个页面；如果使用近似匹配，列出最接近的一个或多个知识点页面，标注为"（近似匹配）"
- **适用依据**：用 1-2 句话解释为什么可以/应该用这个算法类别解这道题——落脚在问题本身的特征上（如"为什么能贪心？因为每步的局部最优选择不会影响后续决策，具备贪心选择性质和无后效性"、"为什么是 DP？因为问题可拆分为重叠子问题且满足最优子结构"）。如果 hello-algo 对应章节中阐述了该算法类别的适用条件或正确性证明，直接引用作为依据，并标注出处（如"见 hello-algo §15.1 贪心算法特性"）
- **核心思路**：用 2-3 句话概括解题策略和正确性依据，通俗解释，不堆砌术语。如果参考了多个知识点，说明它们各自贡献了哪部分思路
- **解题步骤**：与代码逻辑一一对应，读者看完步骤就能对应到代码中的循环/条件分支
- **复杂度**：与代码实现一致的时间/空间复杂度（不照搬理论值）
- **关键洞察**：理解这道题最重要的 1-2 个要点
- **延伸阅读**：hello-algo 链接，以及对应题目来源的官方题解链接（如果知道来源）

**使用用户的语言回复**：如果用户用中文提问就用中文，用英文就用英文。解题思路的语言也跟随用户。

### 第六步：汇总报告

向用户报告：

- 文件路径
- 算法类别 → hello-algo 章节
- 匹配方式：精确匹配 or 近似匹配（说明为什么选择这个知识点）
- 知识点链接
- 新增的内容概要

## 关键规则

1. **信息缺失就问**：无法从文件或上下文中确定的（难度、类别、来源），主动向用户询问，不要猜测
2. **不修改代码**：只添加注释块，不修改任何实现代码。如果函数体为空（尚未实现），仍然可以基于题目描述添加解题思路注释
3. **注释格式适配语言**：根据文件语言使用对应的注释语法（JS/TS/C/Java → `/* */` 或 `//`，Python → `#`，等）
4. **不重复添加**：如果文件中已存在 hello-algo 相关注释块，先询问用户是否覆盖
5. **精确匹配优先，近似匹配兜底**：先尝试在 hello-algo 找到精确专题页；找不到时，自行判断题目类别，从对应章节中找出最接近的知识点（依据解题策略相似度、数据结构重合度、问题模式），可跨章节关联多个知识点。不要在报告中假装找到了精确匹配
6. **语言跟随用户**：用户用什么语言交流就用什么语言回复，解题思路的语言也一致

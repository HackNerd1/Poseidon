# 算法题解优化工作流

## 概述

读取用户提供的算法题文件，分析现有解法的可优化点（代码质量和算法效率两个维度），在文件中新增一个优化版函数（保留原函数），并在注释中标注每个优化点及其原因。

## 核心原则

1. **不修改原函数**：所有涉及代码逻辑/结构的改动都以新函数的形式呈现，函数名在原函数名后加 `Optimized` 后缀。**例外**：纯注释缺失问题（非显而易见的逻辑缺少说明注释）允许直接在原函数中添加注释，无需为此单独创建优化版函数
2. **每个优化点需标注原因**：说明为什么这样改更好，而非仅列出改动
3. **优化维度**：按代码质量和算法效率分层，优先标注影响最大的优化点。代码质量维度重点关注 **边界条件处理**（A5）和 **代码结构**（A6），这两类问题直接影响代码健壮性和可维护性

---

## 工作流

### 第一步：读取题目文件，理解现有解法

读取文件内容，提取：

1. **题目标题与描述**：理解问题域
2. **现有代码实现**：完整的函数逻辑
3. **现有注释中的已知信息**：算法类别、复杂度分析等
4. **编程语言**：确定后续优化策略的适用范围

### 第二步：逐项审查，识别可优化点

按以下清单逐项审查，对每一项给出「优化 / 保留」判断。每个可优化点都需要记录 **具体位置**（变量名、行逻辑、结构）和 **原因**。

#### A. 代码质量维度

| # | 检查项 | 说明 |
|---|-------|------|
| A1 | 变量命名 | 名称是否准确反映内容？（如 Set 被命名为 Map、result 不如 maxLen 语义明确） |
| A2 | 类型标注 | (TS/Java/C++ 等静态类型语言) 泛型参数是否完整？是否依赖隐式 any？ |
| A3 | 中间变量 | 仅用一次的变量是否必要？是否可内联减少认知负担？ |
| A4 | const/let 适用性 | 未重新赋值的变量是否标记为 const？ |
| A5 | ★ 边界条件处理 | 是否可提前返回覆盖空输入/单元素等平凡情况？是否存在未处理的边界导致潜在 bug？—— 这是代码健壮性的底线，需优先审查 |
| A6 | ★ 代码结构 | 条件分支是否可以扁平化？深层嵌套是否可提前 continue/break 或反转条件减少嵌套层级？逻辑相关的代码是否放在邻近位置？ |

#### B. 算法效率维度

| # | 检查项 | 说明 |
|---|-------|------|
| B1 | 数据结构选择 | 当前数据结构是否最优？（Set vs Map vs 数组 — 取决于操作模式：仅判重 vs 需记录位置 vs 定长字符集） |
| B2 | 冗余操作 | 是否有可用数学关系跳过的逐次循环？（如 left 逐个右移 vs 直接跳转） |
| B3 | 循环不变量提取 | 循环内是否有不随迭代变化的值可提升到循环外？ |
| B4 | 字符集特性利用 | 若字符集有限（ASCII/小写字母），是否可用定长数组替代哈希结构？ |

#### C. 注释与可读性

| # | 检查项 | 说明 |
|---|-------|------|
| C1 | 关键逻辑注释 | 非显而易见的逻辑（如 left 跳转条件）是否需要简要说明？ |

### 第三步：生成优化版函数

将优化后的实现写入新函数。要求：

- **函数名**：`<原函数名>Optimized`
- **保留原函数不变**：在原函数下方新增
- **优化版需完整可运行**：与优化点一一对应，是优化建议的落地产物

### 第四步：在优化函数前插入优化说明注释

在优化版函数上方插入注释块，格式如下（根据文件语言适配注释语法）：

```
/**
 * ═══════════════════════════════════════════════
 * 优化版本：<原函数名>Optimized
 * ═══════════════════════════════════════════════
 *
 * 可优化点清单：
 *
 * [代码质量]
 * 1. <优化项标题>
 *    原代码：<具体原代码片段>
 *    优化：<改后代码片段>
 *    原因：<1-2 句解释>
 *
 * 2. ...
 *
 * [算法效率]
 * 1. <优化项标题>
 *    原代码：<具体原代码片段>
 *    优化：<改后代码片段>
 *    原因：<1-2 句解释>
 *
 * 2. ...
 *
 * 复杂度（与原版对比）：
 *   原：时间 O(x) / 空间 O(y)
 *   优化后：时间 O(x) / 空间 O(y)（如有变化则说明）
 * ═══════════════════════════════════════════════
 */
```

### 第五步：汇总报告

向用户报告：

- 文件路径
- 优化点数量（代码质量 N 个 + 算法效率 M 个）
- 每个优化点的标题和一句话原因
- 新增函数名及行号范围

---

## 关键规则

1. **不修改原函数（有一个例外）**：涉及代码逻辑、变量、数据结构的改动必须以新函数呈现，不允许在原函数上直接改。**例外**：仅因注释缺失（关键逻辑无说明注释）时，允许在原函数中直接添加注释，无需为此创建优化版函数
2. **优化点必须附原因**：不写"改名为 xx"，要写"原名 xx 有误导性，因为...，故改为 xx"
3. **复杂度对比**：如果算法效率优化改变了复杂度，必须标注；如果仅改变常数因子，说明即可
4. **语言适配**：注释语法跟随文件语言（TS/JS → `/* */`，Python → `#`，等）
5. **每个优化点独立编号**：方便用户逐一审查是否采纳
6. **如果现有解法无明显优化空间**：如实告知，不强行找优化点
7. **语言跟随用户**：用户用什么语言交流就用什么语言回复

## 示例：常见优化模式

以下为通用优化模式，仅作审查时的参考框架，不针对任何具体题目：

### 模式 1：Set 判重 → Map 记录位置

```
场景：滑动窗口去重时用 while + Set.delete 逐元素移除
问题：每个字符可能被 delete 一次，增加常数操作
改进：用 Map 记录每个字符的最后出现位置，left 直接跳转
效果：从最多 2n 次操作降至恰好 n 次
```

### 模式 2：变量名与实际结构不符

```
场景：const hashMap = new Set()
问题：名称暗示是 Map/HashMap，实际是 Set，误导读者
改进：改为 charSet 或 windowChars
```

### 模式 3：仅用一次的中间变量

```
场景：const element = arr[i];  // 仅下一行使用一次
问题：增加认知负担，不提供额外语义
改进：直接内联 arr[i]
```

### 模式 4：缺少早期返回

```
场景：函数开头无边界检查
问题：空输入或单元素时仍进入完整循环逻辑
改进：if (input.length <= 1) return input.length
```

### 模式 5：泛型参数缺失

```
场景：const s = new Set()  // TypeScript
问题：隐式 any，失去类型检查
改进：new Set<string>()
```
